# JVM Warmup 팀 공유용 1페이지 요약

## 한 문장으로 말하면

`JVM warmup`은 서비스가 이미 떠 있어도 시작 직후에는 아직 내부 최적화와 초기화가 덜 끝나서 초반 성능이 흔들리는 현상이다.

## 팀이 먼저 알아야 할 핵심

- `서비스 기동 완료`와 `성능 안정화 완료`는 다르다.
- 초반 느림은 자바 자체의 실패라기보다 JVM의 적응형 최적화 특성이다.
- 오래 사는 서버는 괜찮을 수 있지만, 서버리스·오토스케일·짧은 워커는 실제 사용자 문제로 이어지기 쉽다.
- steady-state 성능이 좋아도 첫 1분 성능이 나쁘면 운영상 문제다.

## 왜 생기나

- JIT가 자주 실행되는 코드를 나중에 더 강하게 최적화한다.
- 시작 시 클래스 로딩, 프레임워크 초기화, 커넥션 풀 준비가 같이 일어난다.
- 캐시가 비어 있어 첫 요청이 준비 비용을 함께 낸다.
- GC와 메모리 사용 패턴도 초반에는 아직 안정되지 않았다.

## 보통 어떤 식으로 보이나

- 첫 요청이나 첫 수십 초 응답 시간이 길다.
- 시작 직후 CPU가 높게 튄다.
- 같은 API도 몇 분 뒤 처리량이 더 좋아진다.
- 새 인스턴스가 붙을 때마다 tail latency가 흔들린다.

## 팀이 자주 하는 오해

- 초반 느림을 무조건 GC 문제로 본다.
- readiness 통과만 보면 성능 준비도 끝났다고 본다.
- 평균 latency 하나만 보고 판단한다.
- 충분히 데워진 뒤 수치만 보고 cold start 영향을 무시한다.

## 실무에서 바로 해야 할 것

1. `startup time`, `warmup time`, `steady-state performance`를 분리해 본다.
2. 첫 1분의 p95/p99 latency를 별도로 본다.
3. 새 인스턴스에 pre-warming이 필요한지 확인한다.
4. 오토스케일 정책이 startup CPU spike에 과민한지 점검한다.
5. 짧게 사는 워크로드인지, 오래 사는 워크로드인지 먼저 구분한다.

## 대응 우선순위

### 1순위: 측정 방식 고치기

- readiness와 안정화 시점을 분리한다.
- cold, warming, hot 구간을 섞지 않는다.
- 평균값보다 p95/p99와 시간대별 변화를 본다.

### 2순위: 운영 방식 고치기

- pre-warming 트래픽을 넣는다.
- 최소 인스턴스를 유지한다.
- 너무 잦은 재배포, pod churn, scale-to-zero를 줄인다.

### 3순위: startup 비용 줄이기

- 클래스 수와 의존성을 줄인다.
- 불필요한 초기화를 덜어낸다.
- CDS/AppCDS 같은 startup 최적화 수단을 검토한다.

## 팀용 결론

JVM warmup 문제는 `JVM 옵션 미세 조정`부터 시작할 일이 아니라, `초기 성능 구간을 별도 운영 문제로 인정하고 측정/배포/스케일 정책을 함께 손보는 일`로 보는 것이 맞다.
