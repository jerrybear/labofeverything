# JVM Warmup 이슈 한눈에 보기

## 한 줄 정의

`JVM warmup`은 자바 애플리케이션이 시작 직후 바로 최고 성능으로 동작하지 않고, 실행하면서 점차 최적화되어 성능이 안정되는 구간을 말한다.

쉽게 말하면 서비스는 이미 떠 있지만, 내부 엔진이 아직 `예열` 중이라 초반 요청이 더 느릴 수 있는 상태다.

## 대표 증상

- 첫 요청 몇 개의 응답 시간이 평소보다 길다.
- 시작 직후 CPU 사용률이 잠깐 높게 튄다.
- 같은 API인데도 초반과 몇 분 뒤의 처리량이 다르다.
- 오토스케일 환경에서 새 인스턴스가 뜰 때 성능이 들쭉날쭉하다.
- 짧게 살아 있다 죽는 배치, 서버리스, 단명 Pod에서 성능 이점이 충분히 나오지 않는다.

## 왜 생기나

- JVM은 처음부터 모든 코드를 강하게 최적화하지 않는다.
- 먼저 해석 실행 또는 가벼운 컴파일로 빠르게 시작한다.
- 실행 중 자주 호출되는 코드를 관찰해 `hot path`를 찾는다.
- 그 뒤 더 공격적인 최적화를 적용해 점점 빨라진다.
- 이 과정에서 클래스 로딩, 초기화, 캐시 채우기, GC 안정화도 함께 일어난다.

## 구분해서 봐야 하는 세 가지

### 시작 시간

프로세스가 뜨고 헬스체크 기준으로 `준비 완료`가 될 때까지의 시간이다.

### 웜업 시간

서비스는 이미 요청을 받을 수 있지만, 응답 시간과 처리량이 아직 안정되지 않은 시간이다.

### 정상 상태 성능

주요 JIT 최적화와 캐시 초기화가 끝나고 성능이 비교적 안정된 뒤의 상태다.

이 세 가지를 섞어 보면 원인을 잘못 판단하기 쉽다.

## 비전문가 기준으로 이해하는 핵심 포인트

- `애플리케이션이 떴다`와 `최고 성능 상태다`는 같은 말이 아니다.
- 초반 느림이 꼭 장애는 아니지만, 사용자 체감이나 오토스케일에는 실제 문제가 될 수 있다.
- 오래 사는 서버는 웜업 비용을 금방 회수하지만, 짧게 사는 워크로드는 웜업이 본체가 될 수 있다.

## 언제 특히 문제인가

- 서버리스처럼 scale-to-zero가 잦은 환경
- 트래픽 급증 때 새 인스턴스를 자주 추가하는 환경
- 짧게 실행되고 끝나는 배치나 워커
- 첫 응답 체감이 중요한 API
- 인스턴스당 트래픽이 낮아 충분히 뜨거워지기 전에 종료되는 환경

## 언제 상대적으로 덜 문제인가

- 장시간 살아 있는 일반적인 서버 애플리케이션
- 초기 몇 초의 변동보다 장기 처리량이 더 중요한 서비스
- 인스턴스를 오래 유지하고 미리 띄워 둘 수 있는 환경

## 실무에서 자주 하는 오해

- 시작 직후 느리면 무조건 GC 문제라고 보는 것
- 첫 요청 성능만 보고 전체 성능이 나쁘다고 판단하는 것
- 반대로 steady state 숫자만 보고 cold start 문제를 무시하는 것
- 벤치마크에서 cold, warming, hot 상태를 섞어 평균만 보는 것

## 한 줄 결론

JVM warmup 이슈는 `자바가 느리다`의 문제가 아니라, `초기 구간과 정상 구간의 성능 특성이 다르다`는 문제로 이해해야 한다.
