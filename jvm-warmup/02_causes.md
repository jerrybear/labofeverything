# JVM Warmup 이슈의 주요 원인

## 가장 큰 원인: JIT와 프로파일링

JVM은 실행 중 실제 동작을 관찰한 뒤 자주 쓰는 코드를 더 빠른 기계어로 다시 컴파일한다. 이 때문에 시작 직후에는 아직 최적화가 덜 된 코드가 실행되고, 시간이 지나며 성능이 좋아진다.

## Tiered Compilation을 쉽게 보면

- 처음에는 빠르게 시작하는 쪽을 우선한다.
- 이후 자주 호출되는 메서드의 호출 횟수, 분기 패턴, 타입 정보를 모은다.
- 충분히 중요하다고 판단되면 더 무거운 최적화를 적용한다.
- 그래서 초반 응답 시간과 몇 분 뒤 응답 시간이 다를 수 있다.

즉, warmup은 `코드가 점점 더 좋은 버전으로 교체되는 시간`이라고 보면 된다.

## 클래스 로딩과 초기화

JIT만이 전부는 아니다. 애플리케이션이 시작될 때는 다음 작업도 함께 일어난다.

- 필요한 클래스 로딩
- 바이트코드 검증
- static initializer 실행
- 프레임워크 초기화
- DI 컨테이너, 프록시, 리플렉션 관련 준비
- DB 커넥션 풀, HTTP 클라이언트, 내부 캐시 초기화

실무에서는 이 구간이 JIT보다 더 크게 체감될 때도 많다.

## 캐시가 비어 있는 상태

초기에는 여러 계층의 캐시가 비어 있다.

- 애플리케이션 캐시
- JVM 내부 메서드 해석 관련 캐시
- 클래스 메타데이터 관련 공유 영역
- DB 커넥션 풀과 네트워크 연결 상태
- OS 페이지 캐시

같은 요청이라도 첫 번째는 `준비 비용`을 함께 내고, 나중 요청은 이미 채워진 상태를 활용한다.

## GC와 메모리 동작 안정화

시작 직후에는 객체 생성 패턴과 메모리 사용량이 아직 안정되지 않아 GC 동작도 흔들릴 수 있다.

- 초기 객체 생성이 몰릴 수 있다.
- 프레임워크 부팅 중 임시 객체가 많이 생긴다.
- 힙 사용 패턴이 자리 잡기 전까지 pause나 CPU 패턴이 달라질 수 있다.

다만 모든 초반 느림을 GC 탓으로 돌리면 원인을 놓치기 쉽다.

## Deoptimization도 변수다

JIT는 실행 중 세운 가정이 틀리면 기존 최적화를 되돌리고 다시 컴파일할 수 있다. 그래서 warmup은 항상 한 방향으로만 부드럽게 좋아지지 않는다.

- 잠깐 빨라졌다가 다시 흔들릴 수 있다.
- 특정 트래픽 패턴에서 tail latency가 튈 수 있다.
- 컴파일 스레드가 CPU를 쓰면서 요청 처리와 경쟁할 수 있다.

## 왜 환경마다 체감이 다른가

- 인스턴스 수명 길이
- 인스턴스당 트래픽 밀도
- CPU quota와 container 제한
- 프레임워크 초기화 무게
- 클래스 수와 의존성 수
- JVM 버전과 옵션

같은 코드라도 로컬, VM, 컨테이너, 서버리스에서 warmup 체감은 크게 달라질 수 있다.

## 실무적으로 이해하면 좋은 포인트

1. warmup은 한 가지 원인이 아니라 `JIT + 초기화 + 캐시 + 메모리 안정화`의 합이다.
2. startup 문제와 warmup 문제는 겹치지만 같은 문제는 아니다.
3. steady state가 좋아도 초반 사용자 경험은 나쁠 수 있다.
4. 짧게 사는 워크로드에서는 warmup 최적화가 곧 제품 성능 최적화다.

## 한 줄 결론

JVM warmup 이슈의 핵심은 `실행하면서 배우고 최적화하는 JVM의 특성`과 `초기화 비용`이 초반 요청 구간에 함께 겹친다는 점이다.
