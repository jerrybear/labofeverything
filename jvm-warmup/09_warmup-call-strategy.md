# 예열 호출 횟수와 Warmup 전략 정리

## 이 문서의 목적

이 문서는 `대표 API를 몇 번 정도 예열해야 하는가`, `초기화 예열과 JIT 예열은 어떻게 다른가`, `일반적인 애플리케이션 개발에서 이 수준까지 고려해야 하는가`, `JVM 옵션으로 warmup을 없앨 수 있는가`를 한 번에 정리한 문서다.

## 먼저 결론부터

- `1~2번 호출`만으로도 일부 예열 효과는 있다.
- 다만 그것만으로 `JIT warmup`까지 충분히 끝난다고 기대하기는 어렵다.
- 일반적인 웹 애플리케이션에서는 항상 깊게 고려할 필요는 없지만, `오토스케일`, `서버리스`, `짧은 워커`, `첫 응답 민감 서비스`에서는 중요하다.
- JVM 옵션만으로 warmup을 완전히 없애는 것은 보통 어렵고, 일부 구간을 완화하는 정도로 보는 것이 맞다.

## 1~2번 호출로도 효과가 있는가

있다. 특히 `초기화 비용`을 앞당기는 데는 꽤 의미가 있다.

### 1~2번 호출로 잘 줄어드는 것

- DB 커넥션 풀의 첫 연결 비용
- DNS lookup, TLS handshake 같은 외부 연동 초기 연결 비용
- Jackson serializer/deserializer 준비 비용
- 첫 SQL 실행에 붙는 일부 초기 준비 비용
- 캐시 미적재 상태에서 발생하는 첫 비용
- 인증/인가 필터 체인의 첫 통과 비용

즉, `첫 요청에만 유난히 몰리는 one-time cost`를 줄이는 데는 적은 횟수의 호출도 도움이 된다.

## 왜 1~2번으로는 부족할 수 있나

JVM의 `JIT warmup`은 단순히 한 번 호출했다고 끝나는 구조가 아니기 때문이다.

- JVM은 실행 중 호출 횟수, 분기 패턴, 타입 정보 등을 보며 hot path를 판단한다.
- 충분히 자주 실행된 코드만 더 공격적으로 최적화된다.
- 따라서 1~2번 호출은 `초기화 예열`에는 유효해도, `충분한 JIT 최적화`를 만들기에는 약할 수 있다.

즉, `한두 번 태운다`와 `JIT까지 충분히 데운다`는 다른 이야기다.

## 몇 번 호출해야 하나

정답은 고정 숫자보다 목적에 따라 다르게 잡는 것이 맞다.

### 목적이 초기화 예열일 때

- 대표 경로를 `1~2번` 또는 `소수 회수` 호출하는 것으로 시작하는 편이 합리적이다.
- 이 단계에서는 DB 풀, 외부 연결, 직렬화, 캐시, 보안 필터 같은 one-time cost 제거가 핵심이다.

### 목적이 JIT warmup일 때

- 실제 트래픽과 비슷한 요청을 `여러 번` 반복해야 할 수 있다.
- 중요한 것은 횟수 그 자체보다 `실제 hot path를 얼마나 비슷하게 태우는가`다.
- 따라서 운영에서는 `호출 횟수`보다 `첫 1분 p95/p99가 개선됐는지`로 판단하는 편이 더 안전하다.

## 실무적으로 권장하는 시작 방식

### 1단계

- 대표 API 3종을 1~2회씩 호출해 본다.
- cold start 대비 첫 요청 지연이 줄어드는지 본다.

### 2단계

- 효과가 부족하면 해당 경로를 조금 더 반복 호출한다.
- 첫 1분 latency 분포와 startup CPU를 같이 본다.

### 3단계

- 그래도 새 인스턴스 첫 1분 성능이 나쁘면, 호출 횟수보다 CPU throttling, rollout, HPA, min replica 쪽을 함께 점검한다.

즉, 보통은 `적은 호출로 시작 -> 지표 보고 늘리기`가 낫다.

## 비교표 1: 대표 API 예열 횟수 어떻게 정할까

| 목적 | 시작점 | 더 늘려야 하는 신호 | 주의점 |
| --- | --- | --- | --- |
| DB/외부연동/직렬화 초기화 | 대표 경로 1~2회 | 첫 요청만이 아니라 첫 1분 전체가 느림 | 예열 대상이 실제 경로를 반영해야 함 |
| 캐시/인증/필터 초기 준비 | 대표 보호 API 소수 회수 | cold/warm 차이가 여전히 큼 | 헬스체크만 호출하면 효과 약함 |
| JIT warmup 유도 | 실제 hot path를 여러 번 | CPU spike 뒤에도 p99가 안정되지 않음 | 호출만 늘려도 CPU 부하가 커질 수 있음 |
| 전체 서비스 안정화 | 작은 횟수로 시작 후 점진 확대 | 새 Pod 첫 5분 지표가 계속 나쁨 | 호출 횟수보다 rollout/HPA 영향이 더 클 수 있음 |

## 비교표 2: 초기화 예열 vs JIT 예열

| 항목 | 초기화 예열 | JIT 예열 |
| --- | --- | --- |
| 목적 | 첫 요청 one-time cost 제거 | hot path 최적화 유도 |
| 보통 필요한 호출 수 | 적은 편 | 더 많은 편 |
| 대표 대상 | DB 풀, TLS, 직렬화, 캐시, 필터 | 반복 호출되는 비즈니스 경로 |
| 효과가 빨리 보이나 | 비교적 빨리 보임 | 지표로 확인해야 함 |
| 판단 기준 | 첫 요청 지연 감소 | 첫 1분 p95/p99, throughput 안정화 |
| 흔한 실수 | 헬스체크만 호출 | 비현실적 트래픽 패턴으로 예열 |

## 일반적인 애플리케이션 개발에서 이 부분까지 고려해야 하나

항상 깊게 고려해야 하는 것은 아니다. 다만 `문제가 생길 구조인지`는 알고 개발하는 편이 좋다.

### 보통은 크게 안 아픈 경우

- 장시간 살아 있는 일반적인 서버
- 새 인스턴스가 자주 안 붙는 서비스
- 첫 몇 초 응답 편차가 치명적이지 않은 서비스

### 개발 단계부터 고려하는 것이 좋은 경우

- Kubernetes/HPA로 scale out이 잦은 서비스
- CPU limit가 빡빡한 컨테이너 환경
- scale-to-zero나 serverless
- 짧게 실행되고 끝나는 배치/워커
- 첫 응답 체감이 중요한 API

즉, `모든 서비스에서 필수 설계 항목`은 아니지만, 운영 구조상 초반 성능이 중요한 서비스라면 고려하는 것이 맞다.

## JVM 옵션으로 warmup을 없앨 수 있나

완전히 없애기는 어렵다.

이유는 JVM이 실행하면서 실제 동작 패턴을 보고 최적화하는 구조이기 때문이다. 즉, warmup은 단순 설정 부족이 아니라 JVM의 특성에서 오는 부분이 크다.

### 다만 완화는 가능하다

- `CDS/AppCDS`: 클래스 로딩과 startup 비용 완화
- 일부 JIT 관련 옵션 조정: warmup 특성 변화 가능
- AOT 계열 접근: cold start 완화 가능

### 하지만 한계도 있다

- startup은 좋아져도 steady-state trade-off가 생길 수 있다.
- 환경 의존성이 커질 수 있다.
- 운영 복잡도가 올라갈 수 있다.
- 애플리케이션 초기화 비용 자체는 별도로 남을 수 있다.

즉, 옵션은 `보조 수단`이지, 언제나 정답은 아니다.

## 실무 권장 순서

1. 먼저 대표 경로 `1~2회` 수준의 가벼운 pre-warming으로 시작한다.
2. 첫 요청 지연이 줄어드는지 본다.
3. 부족하면 예열 대상을 실제 hot path 기준으로 늘리고 반복 횟수도 늘린다.
4. 그래도 문제면 CPU throttling, rollout 방식, HPA, min replica를 함께 본다.
5. 마지막에 CDS/AppCDS, JVM 옵션, AOT 계열을 검토한다.

## 한 줄 결론

`1~2번 호출`은 보통 `초기화 예열`에는 의미가 있지만, `JIT까지 충분히 데우는 것`은 별개다. 그래서 예열 전략은 고정 횟수보다 `무엇을 예열하려는지`와 `실제 지표가 개선되는지`를 기준으로 잡는 것이 맞다.
