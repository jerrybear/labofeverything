# AI 프롬프트 모음

## 목적

플랫폼 기획자가 반복적으로 수행하는 조사, 정리, 비교, 커뮤니케이션 업무를 빠르게 실행할 수 있도록 바로 사용할 수 있는 프롬프트를 정리한다.

## 사용 원칙

- 민감한 고객 정보, 내부 비밀, 식별 가능한 데이터는 그대로 넣지 않는다.
- 프롬프트 결과는 초안으로 보고, 최종 판단과 대외 공유 전에는 사람이 검토한다.
- 프롬프트는 문서 생산보다 의사결정 품질 향상을 목표로 사용한다.
- 조사형 프롬프트에는 반드시 출처, 근거 수준, 전후 비교 여부를 함께 요구한다.

## 주장 상태와 근거 강도

- 주장 상태: `observed`, `inference`, `hypothesis`, `open question`
- 근거 강도: `strong`, `medium`, `weak`

## 공통 조사 출력 형식

```text
아래 형식으로만 정리해줘.

- 주장:
- 주장 상태: observed / inference / hypothesis / open question
- 직접 근거:
- 인접 근거:
- 수치/기준선:
- 기간:
- 출처:
- 근거 강도: strong / medium / weak
- 해석 주의점:
```

## 데이터 입력 규칙

- 공개 자료와 일반 업무 메모는 우선 허용한다.
- 내부 회의 요약은 식별 정보와 민감 내용 제거 후 사용한다.
- 고객 식별 정보, 계약 정보, 민감 운영 데이터는 승인 없이 넣지 않는다.
- 되돌리기 어려운 정책/보안 판단은 프롬프트 결과만으로 결정하지 않는다.

## 운영 단계 연결

- 신호 수집/정리: 1번 프롬프트
- 문제 정의: 2번 프롬프트
- 국내 사례 검증: 2-1번 프롬프트
- 옵션 설계: 3번 프롬프트
- 우선순위 정리: 4번 프롬프트
- 이해관계자 정렬: 5번 프롬프트
- 실행 후 학습: 6번 프롬프트

## 1. 요청 분류 프롬프트

```text
다음 요청 목록을 플랫폼 관점에서 재분류해줘.

분류 기준:
1) 공통 플랫폼 투자 대상
2) 특정 팀의 로컬 최적화 요청
3) 추가 확인이 필요한 요청
4) 긴급 운영 이슈

각 요청마다 아래 형식으로 정리해줘.
- 요청 요약
- 분류
- 그렇게 판단한 이유
- 추가로 확인할 질문

[요청 목록]
```

## 2. 문제 정의 초안 프롬프트

```text
아래 자료를 바탕으로 플랫폼 기획 관점의 문제 정의 초안을 작성해줘.

포함 항목:
- 문제 상황
- 영향을 받는 사용자/팀
- 현재 손실 또는 비효율
- 추정 원인
- 아직 확인되지 않은 가설
- 지금 내려야 할 의사결정 질문

문서는 짧고 명확하게 작성하고, 증상과 원인을 구분해줘.

[회의 요약 / VOC / 데이터]
```

## 2-1. 국내 사례 검증 프롬프트

```text
아래 국내 사례 자료를 읽고, 플랫폼 기획 효율과의 관련성을 엄격하게 판정해줘.

반드시 아래 항목으로만 답해줘.
- 사례명
- 직접 근거
- 인접 근거
- 전후 수치 존재 여부
- 기간 명시 여부
- 플랫폼 기획과의 거리: 높음 / 중간 / 낮음
- 근거 강도: strong / medium / weak
- 과대해석하면 안 되는 이유

마지막에는 이 사례를 연구 문서에 넣는다면 어떤 한 줄로 요약해야 하는지 써줘.

[기사 / 보도자료 / 인터뷰]
```

## 3. 옵션 비교 프롬프트

```text
아래 문제 정의를 바탕으로 실행 가능한 옵션 3개를 제시해줘.

각 옵션마다 아래 항목을 포함해줘.
- 설명
- 기대 효과
- 비용과 난이도
- 리스크
- 가역성
- 어떤 상황에서 적합한지

마지막에는 표 형태로 3개 옵션을 비교해줘.

[문제 정의]
```

## 4. 우선순위 회의 메모 프롬프트

```text
다음 안건들을 우선순위 회의에 올릴 수 있도록 정리해줘.

정리 형식:
- 안건명
- 안건별 근거 출처
- 기대 영향도
- 확신도
- 비용
- 가역성
- 증거 수준: 강함 / 중간 / 약함
- 추천 우선순위
- 한 줄 근거

마지막에는 회의에서 바로 논의해야 할 쟁점 3개를 뽑아줘.

[안건 목록과 배경]
```

## 5. 이해관계자별 커뮤니케이션 프롬프트

```text
아래 플랫폼 투자안을 서로 다른 이해관계자에게 설명하는 메시지로 바꿔줘.

대상:
1) 경영진
2) 개발팀
3) 운영팀

각 대상별로 아래 형식으로 작성해줘.
- 가장 중요하게 봐야 할 포인트
- 예상 우려
- 전달 메시지 초안

[투자안 요약]
```

## 6. 회고 초안 프롬프트

```text
아래 릴리즈 결과를 바탕으로 플랫폼 기획 회고 초안을 작성해줘.

포함 항목:
- 무엇을 의도했는가
- 실제로 어떤 결과가 나왔는가
- 예상과 달랐던 점
- 유지해야 할 점
- 다음 분기에 수정할 점

마지막에는 다음 액션 후보를 우선순위 순으로 제시해줘.

[릴리즈 결과 / 지표 / 피드백]
```

## 프롬프트 개선 규칙

- 결과가 장황하면 출력 형식을 더 강하게 제한한다.
- 근거가 약하면 반드시 "추가 확인 필요" 항목을 넣는다.
- 같은 업무가 반복되면 팀 공용 템플릿으로 고정한다.
- 프롬프트 변경으로 결론이 달라졌다면 `00_개요.md`와 `07_운영템플릿.md`의 흐름도 함께 점검한다.
- 사례 조사에서는 "효율 향상"이라는 표현만으로 결론 내리지 않고, 어떤 업무가 얼마나 바뀌었는지 분리해 묻는다.
