# AI 오케스트레이션 용어집

이 문서는 앞선 문서들에서 반복해서 등장하는 핵심 용어를 짧고 일관되게 다시 확인할 수 있게 정리한 참고 문서다.

## 먼저 한 문장으로

용어를 정확히 구분하면 도구 비교와 아키텍처 판단이 훨씬 쉬워진다.

## 핵심 용어

### Agent

모델 호출, 도구 사용, 상태 관리 등을 통해 여러 단계를 거쳐 목표를 수행하는 실행 주체다.

### Agent Orchestration

여러 에이전트나 실행 단계를 조정해 누가 무엇을 언제 할지 정하는 계층이다.

### Workflow Orchestration

장기 실행, 재시도, 스케줄링, 승인 대기, 복구처럼 실행 자체의 안정성을 관리하는 계층이다.

### Durable Execution

장애, 재배포, 장시간 대기 상황에서도 상태를 잃지 않고 실행을 이어갈 수 있게 만드는 방식이다.

### Planner-Executor

계획을 세우는 단계와 실제 수행하는 단계를 분리하는 패턴이다.

### Router

입력을 분석해 적절한 에이전트, 모델, 도구, 워크플로로 보내는 분기 패턴이다.

### Reviewer / Critic

생성 결과를 다시 검토하거나 반박해 품질을 높이는 검토 패턴이다.

### Human-in-the-Loop

중요하거나 위험한 단계에 사람 승인, 수정, 최종 판단을 넣는 운영 패턴이다.

### Fan-out / Fan-in

작업을 여러 하위 작업으로 병렬 분산하고, 다시 결과를 모아 합치는 패턴이다.

### Event-Driven Orchestration

이메일, 웹훅, DB 변경, 사용자 요청 같은 이벤트를 시작점으로 흐름이 작동하는 방식이다.

### Checkpoint

실행 중간 상태를 저장해 실패 후 이어서 재개할 수 있게 만드는 저장 지점이다.

### Retry

실패한 작업을 정책에 따라 다시 시도하는 메커니즘이다.

### Resume After Failure

중단된 워크플로를 마지막 안정 지점부터 이어서 다시 실행하는 능력이다.

### Guardrails

위험한 입력, 출력, 도구 사용, 정책 위반 행동을 실행 전에 막는 안전 계층이다.

### Policy / Audit

어떤 행동이 허용되는지 정의하고, 실제 실행 기록을 남겨 책임과 추적 가능성을 확보하는 계층이다.

### Tracing

모델 호출, 툴 사용, 분기, 재시도 같은 실행 경로를 시간 순서대로 보는 관측 방식이다.

### Evaluation

에이전트 결과가 실제로 성공인지 실패인지, 또는 어느 정도 품질인지 판단하는 과정이다.

### Grounding / Faithfulness

응답이 제공된 근거, 검색 결과, 문서 맥락에 충실하게 연결되어 있는지를 보는 개념이다.

### Tool Correctness

에이전트가 올바른 도구를 올바른 인자와 순서로 사용했는지 보는 평가 관점이다.

### Task Completion Rate

에이전트가 실제 사용자 목표를 끝까지 달성한 비율이다.

### Handoff

에이전트가 사람 상담원이나 다른 시스템으로 작업 맥락을 넘기는 과정이다.

### RAG

Retrieval-Augmented Generation의 약자로, 외부 문서나 검색 결과를 불러와 생성 품질을 높이는 방식이다.

### Vector DB

문서나 데이터를 임베딩 형태로 저장하고 유사도 기반 검색을 지원하는 저장소다.

## 자주 헷갈리는 구분

| 헷갈리는 용어 | 구분 포인트 |
| --- | --- |
| agent orchestration vs workflow orchestration | 전자는 누가 무엇을 할지, 후자는 그 실행을 어떻게 안정적으로 끝낼지에 가깝다 |
| tracing vs evaluation | tracing은 무엇이 일어났는지, evaluation은 그것이 잘됐는지를 본다 |
| guardrails vs policy | guardrails는 실행 시점의 차단 장치, policy는 허용 범위를 정하는 기준에 가깝다 |
| agent framework vs workflow engine | 전자는 추론 흐름, 후자는 실행 내구성과 복구에 더 가깝다 |

## 어떻게 활용하면 좋은가

- 문서를 읽다가 용어가 헷갈리면 빠르게 다시 확인하기
- 팀 내 공통 언어를 맞추기
- 도구 비교 전에 문제를 정확히 정의하기

## 결론

AI 오케스트레이션은 용어가 비슷해 보여도 계층과 책임이 다르다. 용어를 정확히 이해하면 설계 판단도 훨씬 쉬워진다.

## 3줄 요약

- 이 문서는 앞선 문서들에서 반복된 핵심 용어를 짧게 다시 확인하는 참고 문서다.
- 용어를 구분하면 프레임워크, 워크플로 엔진, 관측 도구의 역할 차이가 선명해진다.
- 특히 orchestration, durability, tracing, evaluation, guardrails 구분이 중요하다.
