# AI 오케스트레이션 프레임워크 비교

이 문서는 `LangGraph`, `CrewAI`, `AutoGen`을 빠르게 비교해서 어떤 상황에서 무엇을 먼저 보는 게 좋은지 정리한 문서다.

## 먼저 한 줄씩 보면

- `LangGraph`: 상태와 분기를 세밀하게 제어하는 그래프 중심 프레임워크
- `CrewAI`: 역할 기반 멀티에이전트 팀을 빠르게 구성하는 프레임워크
- `AutoGen`: 대화형 멀티에이전트 상호작용을 유연하게 실험하는 프레임워크

## 비교의 핵심 관점

프레임워크를 비교할 때는 단순히 "누가 더 강력한가"보다 아래 기준이 더 중요하다.

1. 흐름 제어를 얼마나 세밀하게 해야 하는가
2. 멀티에이전트를 얼마나 빠르게 구성해야 하는가
3. 장기 실행과 복구를 얼마나 중요하게 보는가
4. 운영 중 추적과 디버깅이 얼마나 중요한가
5. 팀이 선호하는 개발 방식이 코드 중심인지, 역할 중심인지, 대화 중심인지

## 한눈에 보는 비교표

| 항목 | LangGraph | CrewAI | AutoGen |
| --- | --- | --- | --- |
| 기본 사고방식 | 상태 기반 그래프 | 역할 기반 팀 | 대화 기반 에이전트 |
| 잘하는 것 | 복잡한 분기, 순환, 체크포인트 | 빠른 멀티에이전트 구성 | 유연한 상호작용, 협상형 흐름 |
| 강점 | 제어력, 상태 관리, 프로덕션형 흐름 | 이해하기 쉬움, 팀 구조 표현 쉬움 | 대화형 실험, 연구/코딩 시나리오 적합 |
| 약점 | 학습 곡선, 보일러플레이트 | 세밀한 저수준 제어는 제한적 | 장기 운영 안정성은 별도 보강 필요 |
| 잘 맞는 팀 | 제어와 구조를 중시하는 개발팀 | 빠른 도입이 필요한 제품/업무팀 | 실험 중심 연구/프로토타입 팀 |
| 대표 용도 | 리서치 자동화, 승인형 워크플로 | 비즈니스 자동화, 역할 분담형 협업 | 코드 생성, 토론형 에이전트, 실험 |

## 프레임워크별 이해

### LangGraph

#### 어떤 사고방식인가

작업을 노드와 엣지로 구성된 상태 그래프로 본다. 각 단계가 무엇을 하고, 어디로 분기하고, 어떤 상태를 남기는지 직접 설계하는 방식이다.

#### 강점

- 복잡한 분기와 순환 구조를 명확하게 표현할 수 있음
- 상태 기반 제어가 강해서 장기 흐름 설계에 유리함
- 사람 승인, 재시도, 체크포인트 같은 요구와 잘 맞음
- LangChain/LangSmith 생태계와 함께 쓰기 좋음

#### 약점

- 간단한 작업에도 설계 부담이 커질 수 있음
- 추상화보다 제어를 선택한 도구라 초반 진입이 쉽지 않음
- durable execution을 완전히 해결하려면 외부 실행 계층 검토가 필요할 수 있음

#### 가장 잘 맞는 경우

- "언제 어디로 분기하는지"를 명확히 통제해야 할 때
- 사람 승인, 재검토, 재탐색 루프가 중요한 리서치/문서 작성 파이프라인
- 프로토타입보다 운영 흐름의 구조를 먼저 잡아야 할 때

### CrewAI

#### 어떤 사고방식인가

에이전트를 하나의 "팀"으로 보고, 각 에이전트에 역할과 목표를 부여해 협업시키는 방식이다. 리서처, 작성자, 리뷰어 같은 역할 모델이 직관적이다.

#### 강점

- 멀티에이전트 개념을 빠르게 설명하고 구현하기 쉬움
- 역할 기반 협업 구조를 표현하기 좋음
- 비즈니스 자동화나 데모 설계에서 전달력이 좋음
- 빠른 개념 검증과 초기 제품화에 유리함

#### 약점

- 복잡한 상태 전이와 세밀한 분기 제어는 LangGraph보다 덜 직접적임
- 엔터프라이즈 기능을 붙일수록 구조가 커질 수 있음
- 실무에서는 별도 관측/내구성 계층을 함께 설계해야 하는 경우가 많음

#### 가장 잘 맞는 경우

- 역할 중심 워크플로를 빠르게 만들어야 할 때
- 이해관계자에게 "이 에이전트 팀이 어떻게 일하는지" 보여줘야 할 때
- 고객지원, 영업보조, 내부업무 자동화처럼 팀 구조가 자연스러운 경우

### AutoGen

#### 어떤 사고방식인가

에이전트 간 상호작용을 대화로 본다. 각 에이전트가 채팅 참여자처럼 의견을 주고받으며 문제를 해결하는 방식에 가깝다.

#### 강점

- 대화형 멀티에이전트 실험에 매우 유연함
- 코드 생성, 토론, 반복 협업 시나리오와 잘 맞음
- 연구용 실험이나 새로운 패턴 탐색에 적합함

#### 약점

- 흐름이 길어질수록 예측 가능성과 안정성이 떨어질 수 있음
- 상태 지속성과 운영 표준화는 추가 설계가 필요한 편임
- 프로덕션 운영에서는 관측, 제한, 복구 계층이 별도로 필요함

#### 가장 잘 맞는 경우

- 다양한 에이전트가 토론하며 답을 좁혀가는 실험
- 연구형 자동화, 코드 에이전트, 반복 검토형 프로토타입
- 정해진 그래프보다 유연한 상호작용을 우선할 때

## 무엇이 더 프로덕션에 가까운가

상대적으로는 `LangGraph`가 가장 "구조화된 제어"에 가깝고, `CrewAI`는 가장 "빠른 멀티에이전트 도입"에 가깝고, `AutoGen`은 가장 "실험 친화적"이다.

다만 셋 중 하나만으로 운영 안정성이 완성되지는 않는 경우가 많다. 실제로는 아래 계층이 함께 붙는다.

- durable execution: `Temporal`, `Prefect`
- tracing / debugging: `LangSmith`, `Phoenix`, `Langfuse`
- evaluation: `DeepEval` 등
- guardrails / permissions: 정책 엔진, 승인 단계, 툴 권한 통제

## 선택 가이드

### LangGraph를 먼저 볼 때

- 상태와 분기를 설계도로 관리하고 싶다
- 장기 실행 흐름에서 예측 가능성이 중요하다
- 멀티에이전트보다 "흐름 제어"가 더 중요하다

### CrewAI를 먼저 볼 때

- 빠르게 멀티에이전트 개념 검증을 해야 한다
- 역할 중심 협업 구조가 자연스럽다
- 비기술 이해관계자에게 구조를 설명하기 쉬워야 한다

### AutoGen을 먼저 볼 때

- 대화형 실험과 반복 협업이 핵심이다
- 연구/코드 생성/토론형 시나리오가 많다
- 더 자유로운 상호작용을 먼저 시험해보고 싶다

## 실무적으로는 이렇게 고르면 된다

| 상황 | 먼저 볼 도구 | 이유 |
| --- | --- | --- |
| 복잡한 승인/재검토/분기 흐름 | LangGraph | 흐름 제어와 상태 관리가 핵심이기 때문 |
| 빠른 멀티에이전트 데모 | CrewAI | 역할 기반 팀 구성이 빠르기 때문 |
| 연구형 토론 에이전트 | AutoGen | 대화 기반 협업 모델이 자연스럽기 때문 |
| 장시간 실행과 복구까지 중요 | LangGraph 또는 CrewAI + Temporal/Prefect | 에이전트 프레임워크만으로는 부족할 수 있기 때문 |

## 흔한 실수

- 멀티에이전트가 필요 없는 문제에 프레임워크부터 복잡하게 도입함
- 빠른 데모 성공을 운영 안정성으로 착각함
- tracing과 evaluation 없이 프레임워크만 비교함
- 에이전트 계층 문제와 워크플로 계층 문제를 섞어서 판단함

## 결론

세 도구는 경쟁 관계이기도 하지만, 더 정확히는 서로 다른 사고방식을 대표한다.

- `LangGraph`는 제어 중심
- `CrewAI`는 역할 중심
- `AutoGen`은 대화 중심

그래서 선택은 "가장 인기 있는 것"보다 "내가 다루려는 문제를 어떤 구조로 표현할 것인가"에 따라 달라진다.

- `LangGraph`는 제어 중심, `CrewAI`는 역할 중심, `AutoGen`은 대화 중심이라는 차이로 보면 이해가 쉽다.
- 셋 중 무엇을 고르든 운영 환경에서는 durable execution, tracing, evaluation 계층을 함께 검토해야 한다.
- 프레임워크 선택은 인기보다 문제를 어떤 구조로 표현할 것인가에 따라 달라진다.
