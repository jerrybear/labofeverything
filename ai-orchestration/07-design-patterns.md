# AI 오케스트레이션 디자인 패턴

이 문서는 실무에서 자주 재사용되는 AI 오케스트레이션 패턴을 모아, 어떤 문제에 어떤 흐름을 적용하면 좋은지 정리한 문서다.

## 먼저 한 문장으로

오케스트레이션 패턴은 에이전트를 많이 붙이는 기술이 아니라, 문제의 구조에 맞게 실행 흐름을 설계하는 방법이다.

## 왜 패턴으로 봐야 하나

같은 도구를 써도 흐름을 어떻게 짜느냐에 따라 결과가 크게 달라진다. 실제 차이는 프레임워크 이름보다 아래에 더 가깝다.

- 누가 먼저 결정하는가
- 누가 실행하는가
- 누가 검토하는가
- 언제 사람에게 넘기는가
- 무엇을 병렬로 처리하는가

즉, 도구보다 먼저 "어떤 패턴으로 묶을 것인가"를 보는 편이 실전에서는 더 유용하다.

## 패턴을 고를 때 먼저 볼 질문

1. 작업이 단계적으로 분해 가능한가?
2. 입력 종류에 따라 다른 경로가 필요한가?
3. 품질 검토 루프가 필요한가?
4. 사람 승인이 필요한가?
5. 동시에 처리 가능한 하위 작업이 많은가?
6. 이벤트가 들어올 때마다 자동으로 반응해야 하는가?

## 패턴 1: Planner-Executor

### 구조

플래너가 해야 할 일을 나누고, 익스큐터가 각 단계를 실제로 수행하는 방식이다.

### 잘 맞는 경우

- 리서치 자동화
- 문서 작성 파이프라인
- 다단계 코드 작업

### 장점

- 계획과 실행 책임이 분리되어 흐름이 명확해진다
- 긴 작업을 단계별로 나누기 쉬워진다
- 실패한 단계를 다시 실행하기가 상대적으로 수월하다

### 주의할 점

- 플래너가 지나치게 세세하면 오히려 비효율적이다
- 계획이 현실과 어긋나면 전체 흐름이 흔들린다

## 패턴 2: Router

### 구조

입력을 분류해서 적절한 에이전트, 모델, 워크플로로 보내는 방식이다.

### 잘 맞는 경우

- 고객지원 문의 분류
- 여러 유형의 문서 처리
- 태스크 성격에 따라 모델을 다르게 쓰는 경우

### 장점

- 입력에 맞는 전문 경로를 사용할 수 있다
- 비용과 성능을 균형 있게 조절하기 좋다

### 주의할 점

- 라우터 정확도가 낮으면 뒤 단계가 모두 어긋난다
- 분기 수가 많아지면 유지보수가 어려워질 수 있다

## 패턴 3: Reviewer / Critic

### 구조

생성하는 주체와 검토하는 주체를 나눠, 결과를 재검토하거나 보완하는 패턴이다.

### 잘 맞는 경우

- 요약문 품질 검토
- 코드 생성 후 리뷰
- 리서치 결과의 근거 확인

### 장점

- 품질 향상에 직접적이다
- 잘못된 결과를 초기에 걸러낼 수 있다

### 주의할 점

- 검토 루프가 길어지면 지연시간과 비용이 커진다
- 비판만 하고 종료 조건이 없으면 무한 반복 위험이 있다

## 패턴 4: Human-in-the-Loop

### 구조

위험하거나 중요한 단계에서 사람 승인, 수정, 최종 판단을 넣는 패턴이다.

### 잘 맞는 경우

- 환불, 결제, 배포 승인
- 법무/규제 문서 검토
- 고객에게 직접 나가는 민감 응답

### 장점

- 위험 통제와 책임 분리가 쉬워진다
- 초반 운영에서 품질 기준을 만들기 좋다

### 주의할 점

- 사람 대기 시간이 병목이 될 수 있다
- durable execution 없이 붙이면 대기 상태 관리가 어렵다

## 패턴 5: Parallel Fan-out / Fan-in

### 구조

하나의 작업을 여러 하위 작업으로 병렬 분산한 뒤, 결과를 다시 합치는 방식이다.

### 잘 맞는 경우

- 여러 문서 동시 분석
- 후보안 여러 개 생성 후 비교
- 멀티소스 리서치

### 장점

- 속도를 높이기 좋다
- 독립적인 하위 태스크에 특히 강하다

### 주의할 점

- 병렬화가 항상 효율적인 것은 아니다
- 집계 단계에서 충돌이나 중복이 생길 수 있다

## 패턴 6: Event-Driven Orchestration

### 구조

사용자 요청, 이메일, 슬랙 메시지, DB 변경, 웹훅 같은 이벤트를 기준으로 워크플로가 시작되는 패턴이다.

### 잘 맞는 경우

- 내부 업무 자동화
- 운영 알림 기반 처리
- SaaS 연동 자동화

### 장점

- 실시간성 있는 업무 자동화에 적합하다
- 기존 시스템과 연결하기 좋다

### 주의할 점

- 이벤트가 많아지면 중복 처리와 루프 문제가 생길 수 있다
- 멱등성과 재시도 설계가 중요해진다

## 패턴은 보통 조합해서 쓴다

실무에서는 하나만 단독으로 쓰기보다 조합이 일반적이다.

- router + planner-executor: 입력별로 다른 계획을 세우는 구조
- planner-executor + reviewer: 실행 후 품질 검토를 붙이는 구조
- event-driven + HITL: 이벤트로 시작하지만 중요한 단계는 사람 승인으로 넘기는 구조
- parallel fan-out/fan-in + reviewer: 병렬 수집 후 품질 비교를 붙이는 구조

## 아주 거칠게 고르는 기준

| 문제 성격 | 먼저 떠올릴 패턴 |
| --- | --- |
| 복잡한 다단계 작업 | planner-executor |
| 입력 유형이 다양함 | router |
| 품질 검토가 중요함 | reviewer / critic |
| 리스크가 높음 | human-in-the-loop |
| 독립 하위 작업이 많음 | parallel fan-out / fan-in |
| 외부 이벤트에 반응해야 함 | event-driven |

## 흔한 실수

- 필요 이상으로 멀티에이전트 구조를 복잡하게 만듦
- reviewer를 넣고도 종료 조건을 정의하지 않음
- 라우터 품질을 검증하지 않고 뒤 단계에만 집중함
- HITL을 붙이면서 대기/재개 구조를 설계하지 않음
- 병렬화했지만 집계 단계 설계를 소홀히 함

## 결론

좋은 오케스트레이션 설계는 많은 패턴을 쓰는 것이 아니라, 문제 구조에 맞는 최소 패턴을 정확히 고르는 데서 시작한다.

## 3줄 요약

- 디자인 패턴은 도구 선택보다 먼저, 문제를 어떤 흐름으로 표현할지 정하는 틀이다.
- planner, router, reviewer, HITL, parallel, event-driven 패턴이 실무에서 가장 자주 반복된다.
- 실제 설계는 단일 패턴보다 여러 패턴의 조합으로 완성되는 경우가 많다.
