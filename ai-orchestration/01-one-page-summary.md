# AI 오케스트레이션 1페이지 요약

## 먼저 한 문장으로

AI 오케스트레이션은 모델을 잘 고르는 문제가 아니라, 여러 모델과 도구와 업무 단계를 연결해서 실제 일을 끝까지 수행하게 만드는 설계 문제다.

## 왜 갑자기 중요해졌나

LLM 하나에 프롬프트를 넣어 답변만 받는 방식으로는 실무 자동화가 잘 끝나지 않는다. 실제 업무는 검색, 도구 호출, 여러 단계 추론, 실패 복구, 사람 승인, 로그와 평가가 함께 필요하다. 그래서 "모델 성능"보다 "흐름 제어와 운영"이 더 중요해졌다.

## 이 문서에서 기억할 핵심 5가지

1. 좋은 모델만으로는 부족하다. 연결, 제어, 복구, 관측이 있어야 한다.
2. 에이전트 오케스트레이션과 워크플로 오케스트레이션은 비슷해 보여도 역할이 다르다.
3. 멀티에이전트는 강력하지만, 디버깅과 비용이 더 어려워진다.
4. 프로덕션에서는 durable execution, tracing, evaluation이 거의 필수다.
5. 도구 선택은 "얼마나 똑똑한가"보다 "어떤 업무를 얼마나 안정적으로 끝내는가"로 봐야 한다.

## 가장 쉬운 구분

### 1. LLM 앱

프롬프트를 보내고, 검색을 붙이고, 툴을 호출하는 애플리케이션 층이다.

- 예: LangChain, LlamaIndex, Vercel AI SDK
- 초점: 모델 호출, RAG, 메모리, 툴 연결

### 2. 에이전트 오케스트레이션

누가 무엇을 먼저 할지, 언제 분기할지, 어떤 역할로 협업할지를 정하는 층이다.

- 예: LangGraph, CrewAI, AutoGen
- 초점: 계획, 라우팅, 멀티에이전트, 상태 기반 흐름

### 3. 워크플로 오케스트레이션

오래 걸리는 작업을 중간에 안 잃고 끝까지 돌리기 위한 실행 인프라 층이다.

- 예: Temporal, Prefect, Airflow
- 초점: 재시도, 체크포인트, 스케줄링, 장기 실행, 장애 복구

### 4. 운영 계층

이 시스템이 실제로 잘 돌아가는지 보는 층이다.

- 예: LangSmith, Phoenix, Langfuse, DeepEval
- 초점: tracing, evaluation, 품질 관리, 비용/성능 관측

## 자주 보이는 오해

### "AI 오케스트레이션 = 멀티에이전트"

아니다. 단일 에이전트나 단일 앱도 충분히 오케스트레이션 대상이 된다. 핵심은 여러 단계를 조정하고 운영하는 것이다.

### "LangGraph 같은 프레임워크 하나면 다 된다"

아니다. 복잡한 실무에서는 종종 에이전트 프레임워크 위에 Temporal 같은 durable workflow 엔진이 추가된다.

### "에이전트를 많이 나누면 더 똑똑해진다"

항상 그렇지 않다. 역할 분담은 좋아질 수 있지만, 상태 동기화, 비용, 실패 원인 분석은 더 어려워진다.

### "평가는 나중에 붙이면 된다"

실제로는 초반부터 tracing과 평가 기준이 없으면 무엇이 좋아졌는지 알기 어렵다.

## 가장 흔한 실무 구조

### 구조 A: 앱 안에서 간단히 연결

- 구성: 앱 서버 + 모델 + 검색/툴
- 장점: 빠르게 시작 가능
- 한계: 장기 실행과 장애 복구가 약함

### 구조 B: 에이전트 프레임워크 중심

- 구성: LangGraph/CrewAI/AutoGen + 모델 + 관측 도구
- 장점: 분기, 역할 분담, 상태 흐름을 다루기 좋음
- 한계: 실행 내구성은 별도 고려가 필요함

### 구조 C: 에이전트 + durable workflow

- 구성: 에이전트 프레임워크 + Temporal/Prefect + 관측/평가
- 장점: 장시간 작업, 승인 대기, 재시도, 복구에 강함
- 한계: 설계와 운영 난도가 올라감

## 도구를 아주 거칠게 고르는 기준

- 빠른 개념 검증이면: CrewAI, n8n 같은 도구가 이해가 쉬움
- 복잡한 상태 흐름이면: LangGraph가 유리함
- 대화형 멀티에이전트 실험이면: AutoGen이 잘 맞음
- 장기 실행과 복구가 중요하면: Temporal, Prefect를 함께 봐야 함
- 데이터 파이프라인 중심이면: Airflow, Prefect가 자연스러움
- 운영 품질이 중요하면: LangSmith, Phoenix, DeepEval을 초반부터 붙여야 함

## 결국 무엇을 보는 분야인가

AI 오케스트레이션은 "좋은 답변 생성"보다 "복잡한 일을 안정적으로 완료"하는 데 더 가깝다. 그래서 앞으로 이 분야를 볼 때는 모델 자체보다 다음 질문이 더 중요하다.

- 어떤 단계를 누가 수행하는가?
- 실패하면 어디서 다시 시작하는가?
- 사람 승인은 어디에 들어가는가?
- 상태는 어떻게 저장되는가?
- 결과 품질은 어떻게 측정하는가?
