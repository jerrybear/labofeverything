# 스프링 배치 한눈에 보기

## 한 줄 정의

스프링 배치는 `대량의 데이터를 정해진 순서대로 안정적으로 처리하는 자바 기반 배치 작업 프레임워크`다.

쉽게 말하면, 사람이 매일 반복해서 하거나 시스템이 밤마다 몰아서 해야 하는 일을 `실수 적게`, `다시 시작 가능하게`, `기록이 남게` 처리하도록 도와주는 도구다.

## 왜 쓰나

- 수만 건에서 수백만 건까지 한꺼번에 처리하는 작업이 많다.
- 중간에 실패해도 처음부터 전부 다시 하지 않고 이어서 처리하고 싶다.
- 실패 기록, 성공 기록, 처리 건수 같은 운영 정보가 필요하다.
- 파일 읽기, 데이터 변환, 데이터베이스 저장 같은 반복 패턴을 표준화하고 싶다.
- 팀마다 제각각 배치 코드를 만들지 않고 검증된 방식으로 만들고 싶다.

## 대표적으로 하는 일

- CSV, 엑셀, JSON, XML 같은 파일을 읽어 DB에 적재
- 여러 시스템의 데이터를 밤마다 동기화
- 정산, 집계, 청구, 리포트 생성 같은 마감성 작업 수행
- 대량 회원 데이터 정제, 마이그레이션, 백필 작업 수행
- 오래된 데이터 정리, 상태 전환, 통계 테이블 갱신

## 비전문가 기준으로 이해하는 핵심 포인트

- 스프링 배치는 `웹 요청을 바로 응답하는 프레임워크`가 아니라 `뒤에서 묵직한 일을 처리하는 프레임워크`에 가깝다.
- `언제 실행할지`를 정하는 도구라기보다 `실행되었을 때 어떻게 안전하게 처리할지`를 맡는 도구다.
- 즉, `스케줄러`가 일을 시작시키고 `스프링 배치`가 실제 일을 처리한다고 보면 이해하기 쉽다.

## 왜 실무에서 선호되나

- 실패 복구 기능이 강하다.
- 대량 데이터 처리 패턴이 잘 정리돼 있다.
- 스프링 생태계와 잘 맞는다.
- 운영 중 문제 추적이 비교적 쉽다.
- 단순 반복문보다 구조가 명확해 협업에 유리하다.

## 장점

- 재시작, 재시도, 건너뛰기 같은 운영 기능이 기본 제공된다.
- 읽기, 가공, 저장 구조가 표준화돼 유지보수가 쉽다.
- 트랜잭션 관리와 처리 이력 관리가 강하다.
- 병렬 처리나 분할 처리로 확장할 수 있다.

## 단점

- 처음 배울 때 개념이 다소 많다.
- 작은 작업에는 설정이 무겁게 느껴질 수 있다.
- 배치 메타데이터 저장소와 운영 기준을 이해해야 한다.
- 단순 스크립트로 끝날 일을 너무 크게 만들 위험이 있다.

## 언제 특히 잘 맞나

- 하루 한 번, 매시간, 매주처럼 반복 실행되는 업무
- 대량 데이터 처리와 실패 복구가 중요한 업무
- 사람이 엑셀로 하던 후처리를 시스템화하려는 경우
- 처리 결과를 감사 가능하게 남겨야 하는 경우

## 언제 굳이 안 써도 되나

- 데이터가 아주 적고 일회성인 경우
- 즉시 응답이 중요한 실시간 API 처리인 경우
- 로직이 너무 단순해서 일반 서비스 코드나 짧은 스크립트로 충분한 경우

## 지금 시점 메모

- 공식 문서 기준 최신 안정 버전 라인은 `6.0.3`이다.
- 다만 실제 프로젝트에서는 `Spring Boot`가 관리하는 버전 조합을 함께 확인해야 한다.
- 새 프로젝트는 보통 자바 기반 설정을 사용하고, 오래된 XML 중심 예제는 그대로 복사하지 않는 편이 안전하다.

## 한 줄 결론

스프링 배치는 `대량 데이터 처리`, `실패 복구`, `반복 업무 자동화`가 중요한 환경에서 가장 빛나는 표준 도구다.
