# 스프링 배치 면접 대비 Q&A

## 이 문서 사용법

- 먼저 질문만 보고 30초 안에 답해 본다.
- 그다음 모범 답안을 읽는다.
- 마지막에는 `내 프로젝트 예시`를 한 줄 붙이면 훨씬 강한 답변이 된다.

## 1. 스프링 배치가 무엇인가요

스프링 배치는 대량 데이터를 안정적으로 처리하기 위한 배치 프레임워크입니다. 읽기, 가공, 저장 흐름을 표준화하고, 재시작, retry, skip, 메타데이터 관리 같은 운영 기능을 제공합니다.

## 2. Job과 Step의 차이는 무엇인가요

Job은 배치 작업 전체 단위이고, Step은 그 안의 개별 처리 단계입니다. 보통 하나의 Job 안에 여러 Step을 두고 순차적으로 실행합니다.

## 3. ItemReader, ItemProcessor, ItemWriter의 역할은 무엇인가요

- Reader는 데이터를 읽고
- Processor는 데이터를 검증하거나 변환하고
- Writer는 최종 결과를 저장합니다.

이 구조 덕분에 책임 분리가 명확해지고 테스트와 유지보수가 쉬워집니다.

## 4. chunk 방식과 tasklet 방식의 차이는 무엇인가요

chunk는 대량 데이터를 일정 개수씩 읽고 처리하고 저장하는 방식이라 데이터 중심 작업에 적합합니다. tasklet은 파일 삭제나 저장 프로시저 호출처럼 단일 작업성 로직에 적합합니다.

## 5. JobParameters는 왜 중요한가요

JobParameters는 실행 시점 입력값이며 JobInstance를 구분하는 기준이 됩니다. 예를 들어 처리일자나 파일 경로를 파라미터로 주면 같은 Job이라도 서로 다른 실행으로 구분할 수 있습니다.

## 6. JobInstance와 JobExecution 차이를 설명해 주세요

JobInstance는 같은 Job과 같은 파라미터 조합으로 정의되는 논리적 실행 단위입니다. JobExecution은 그 Instance에 대한 실제 실행 시도 기록입니다. 한 번 실패 후 재실행하면 JobExecution은 여러 개가 될 수 있습니다.

## 7. 스프링 배치의 restartability는 어떻게 동작하나요

실행 중간 상태를 메타데이터와 ExecutionContext에 저장하고, 재실행 시 저장된 상태를 바탕으로 이어서 처리할 수 있습니다. 그래서 긴 배치 작업이 실패해도 처음부터 다시 하지 않아도 됩니다.

## 8. ExecutionContext는 무엇인가요

Step이나 Job 실행 중 필요한 상태값을 저장하는 영속성 있는 컨텍스트입니다. 예를 들면 마지막으로 읽은 위치나 처리 건수 같은 정보를 저장해 재시작에 활용할 수 있습니다.

## 9. skip과 retry의 차이는 무엇인가요

skip은 문제 있는 데이터를 건너뛰고 계속 처리하는 것이고, retry는 일시적 오류에 대해 같은 작업을 다시 시도하는 것입니다. 데이터 품질 이슈에는 skip이, 네트워크 타임아웃이나 DB 락 같은 일시 장애에는 retry가 더 잘 맞습니다.

## 10. 청크 단위 트랜잭션은 왜 중요한가요

청크 단위로 커밋하면 성능과 안정성의 균형을 잡기 좋습니다. 너무 자주 커밋하면 비효율적이고, 너무 크게 잡으면 실패 시 롤백 비용이 커질 수 있습니다.

## 11. 스프링 배치는 스케줄러인가요

아닙니다. 스프링 배치는 배치 처리 프레임워크이고, 언제 실행할지는 Quartz, cron, `@Scheduled`, Kubernetes CronJob 같은 스케줄러가 담당하는 경우가 많습니다.

## 12. JobRepository의 역할은 무엇인가요

JobRepository는 JobExecution, StepExecution, ExecutionContext 같은 메타데이터를 저장하고 관리합니다. 이 정보가 있어야 실행 이력 추적, 재시작, 운영 모니터링이 가능합니다.

## 13. 메타데이터 테이블은 왜 필요한가요

배치가 언제 실행됐고 어디서 실패했는지, 몇 건 처리했는지를 기록하기 위해 필요합니다. 단순 로그보다 구조화된 운영 추적이 가능하다는 점이 큽니다.

## 14. idempotency를 왜 신경 써야 하나요

배치는 재실행 가능성이 높기 때문입니다. 같은 Job이 다시 실행돼도 중복 적재나 중복 처리 문제가 없도록 설계해야 운영 사고를 줄일 수 있습니다.

## 15. 스프링 배치에서 성능을 높이는 방법은 무엇이 있나요

청크 크기 조정, 병렬 Step, 멀티스레드 Step, 파티셔닝, I/O 병목 개선 같은 방법이 있습니다. 다만 성능 최적화보다 먼저 정합성과 재처리 전략이 맞는지 확인해야 합니다.

## 16. 스프링 배치를 언제 쓰지 않는 게 좋나요

실시간 응답이 중요한 API 처리, 아주 단순한 일회성 스크립트, 소량 데이터 처리에는 과할 수 있습니다. 이런 경우 일반 서비스 로직이나 간단한 CLI가 더 적합할 수 있습니다.

## 17. 실무에서 가장 자주 묻는 설계 포인트는 무엇인가요

대개 아래 네 가지입니다.

- 어떤 기준으로 JobParameters를 잡았는가
- 실패 시 재시작 전략은 무엇인가
- skip/retry 정책은 왜 그렇게 정했는가
- 중복 처리 방지는 어떻게 했는가

면접에서는 개념 설명보다 `왜 그렇게 설계했는지`를 같이 말하면 훨씬 좋습니다.

## 18. 마지막으로 한 문장으로 설명해 보세요

스프링 배치는 대량 데이터를 운영 가능한 방식으로 처리하기 위해 읽기, 가공, 저장, 재시작, 오류 대응, 실행 이력 관리를 체계화한 프레임워크입니다.

## 짧은 면접 팁

- `스케줄링`과 `배치 처리`를 분리해서 설명하면 좋다.
- `chunk vs tasklet`은 거의 자주 나온다.
- `JobInstance vs JobExecution`을 헷갈리지 않으면 인상이 좋다.
- 가능하면 본인 프로젝트에서 처리량, 장애 대응, 재처리 경험을 함께 말한다.
