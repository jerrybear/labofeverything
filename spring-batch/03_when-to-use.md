# 언제 쓰고 언제 말아야 하나

## 잘 맞는 상황

### 1. 대량 데이터를 정기적으로 처리할 때

- 매일 정산
- 매시간 집계
- 야간 데이터 동기화
- 대용량 파일 적재

### 2. 실패 복구가 중요할 때

- 2시간 돌던 작업이 95퍼센트 지점에서 실패할 수 있다.
- 이런 경우 처음부터 다시 하는 비용이 크다.
- 스프링 배치는 재시작 설계에 강하다.

### 3. 운영 추적이 필요할 때

- 언제 시작했는지
- 몇 건 성공했는지
- 어떤 예외가 났는지
- 어디에서 실패했는지

이런 기록이 필요한 환경에 잘 맞는다.

### 4. 팀 표준화가 필요할 때

- 개발자마다 제각각 배치 코드를 짜면 운영 품질이 흔들린다.
- 스프링 배치는 구조를 표준화해 협업과 인수인계에 유리하다.

## 애매하거나 과할 수 있는 상황

### 1. 일회성 데이터 수정

- 오늘 한 번만 돌리고 끝나는 스크립트라면 과할 수 있다.

### 2. 소량 데이터 처리

- 수십 건, 수백 건 수준이면 일반 서비스 로직이나 간단한 CLI로 충분할 수 있다.

### 3. 실시간 응답이 중요한 기능

- 사용자가 버튼을 누르면 즉시 끝나야 하는 기능에는 일반 API 설계가 더 맞다.

### 4. 배치 메타데이터 운영이 부담인 팀

- 단순히 코드만 짜면 끝나는 것이 아니라 실행 기록과 운영 방식도 함께 관리해야 한다.

## 장점 요약

- 실패 복구에 강하다.
- 대량 처리 패턴이 잘 정리돼 있다.
- 스프링 생태계와 통합이 쉽다.
- 테스트와 운영 관점의 도구가 많다.
- 확장 구조가 비교적 명확하다.

## 단점 요약

- 초기 학습 비용이 있다.
- 작은 작업에는 무겁다.
- 개념을 잘못 이해하면 오히려 복잡해진다.
- 청크 크기, 병렬 처리, 예외 정책 같은 운영 판단이 필요하다.

## 빠른 판단 체크리스트

아래 항목 중 여러 개가 맞으면 도입 가치가 높다.

- 처리 건수가 많다.
- 반복 실행된다.
- 실패 후 재시작이 중요하다.
- 성공/실패 이력을 남겨야 한다.
- 읽기/가공/저장 흐름이 분명하다.
- 향후 데이터 규모가 더 커질 가능성이 있다.

반대로 아래가 많으면 굳이 안 써도 된다.

- 일회성이다.
- 데이터가 적다.
- 즉시 응답이 중요하다.
- 운영 기록보다 단순성이 더 중요하다.

## 추천 판단 기준

`배치 운영 문제`가 이미 보이기 시작했다면 스프링 배치를 검토할 시점이다.

예를 들면:

- 수동 재처리가 잦다
- 실패 원인을 추적하기 어렵다
- 야간 작업 시간이 계속 늘어난다
- 작업별 책임 구분이 안 된다

이런 신호가 있으면 단순 스크립트보다 프레임워크 도입 효과가 커진다.

## 한 줄 결론

스프링 배치는 `단순 코드 작성 도구`라기보다 `반복 대량 처리 업무를 운영 가능한 형태로 만드는 도구`에 가깝다.
