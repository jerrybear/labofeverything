# 스프링 배치 핵심 개념

## 큰 그림

스프링 배치는 보통 아래 흐름으로 이해하면 쉽다.

```text
Job
  -> Step 1
  -> Step 2
  -> Step 3
```

- `Job`: 전체 배치 작업 한 덩어리
- `Step`: 그 안의 개별 처리 단계

예를 들어 `회원 휴면 전환 배치`라는 Job 안에 아래 Step이 있을 수 있다.

1. 대상 회원 조회
2. 조건 검증 및 상태 변경
3. 결과 저장 및 알림 로그 적재

## Reader / Processor / Writer

가장 대표적인 처리 방식은 아래 구조다.

```text
읽기 -> 가공 -> 저장
Reader -> Processor -> Writer
```

- `ItemReader`: 데이터를 읽는다.
- `ItemProcessor`: 읽은 데이터를 가공하거나 검증한다.
- `ItemWriter`: 최종 결과를 저장한다.

예시:

- Reader: CSV 파일에서 회원 데이터 읽기
- Processor: 잘못된 이메일 제거, 상태값 변환
- Writer: DB에 저장

## 청크 처리란 무엇인가

청크 처리는 데이터를 `한 건씩 읽되`, `일정 개수씩 묶어서 저장`하는 방식이다.

예를 들어 청크 크기를 100으로 잡으면:

1. 100건 읽기
2. 100건 가공
3. 100건 저장
4. 커밋
5. 다음 100건 반복

이 방식의 장점:

- 메모리를 덜 무리하게 쓴다.
- 너무 자주 커밋하지 않아 성능이 좋아질 수 있다.
- 실패 시 어느 지점까지 처리됐는지 관리하기 쉽다.

## Tasklet은 언제 쓰나

모든 작업이 `읽기 -> 가공 -> 저장` 구조인 것은 아니다.

이럴 때는 `Tasklet`을 쓴다.

예시:

- 특정 폴더 파일 삭제
- 저장 프로시저 한 번 호출
- 임시 테이블 비우기
- 배치 시작 전 준비 작업

즉, `데이터 묶음 처리`에는 청크가 잘 맞고, `단일 작업`에는 Tasklet이 잘 맞는다.

## Job, JobInstance, JobExecution

실무에서 헷갈리기 쉬운 개념이다.

- `Job`: 작업 정의 자체
- `JobInstance`: 특정 파라미터로 구분되는 논리적 실행 단위
- `JobExecution`: 실제 실행 시도 기록

예를 들어 `2026-03-28 정산 배치`가 한 번 실패하고 다시 성공했다면:

- Job은 1개
- JobInstance도 1개
- JobExecution은 2개일 수 있다

이 구조 덕분에 `같은 작업을 다시 돌린 것인지`, `새 파라미터로 다른 작업을 돌린 것인지`를 구분할 수 있다.

## 재시작과 ExecutionContext

스프링 배치의 강점 중 하나는 `중간 실패 후 재시작`이다.

- 어디까지 처리했는지 상태를 저장할 수 있다.
- 재실행 시 처음부터 전부 다시 하지 않고 이어서 갈 수 있다.
- 긴 작업일수록 이 기능의 가치가 커진다.

이때 상태를 저장하는 데 핵심이 되는 개념이 `ExecutionContext`다.

쉽게 말하면 `현재 진행 상황을 적어 두는 작업 메모장`에 가깝다.

## skip 과 retry

둘 다 장애 대응이지만 목적이 다르다.

### skip

문제가 있는 데이터를 `건너뛰고 계속 진행`하는 방식이다.

예시:

- CSV 10만 건 중 3건 형식 오류
- 오류 3건은 로그에 남기고 나머지는 계속 처리

잘 맞는 경우:

- 일부 불량 데이터가 전체 작업을 막으면 안 되는 경우

주의할 점:

- 금융 정산처럼 한 건도 틀리면 안 되는 업무에서는 신중해야 한다.

### retry

일시적 실패를 `다시 시도`하는 방식이다.

예시:

- 일시적 DB 락
- 잠깐 끊긴 외부 API 호출
- 네트워크 타임아웃

잘 맞는 경우:

- 잠깐 후 다시 하면 성공할 가능성이 큰 장애

## 스케줄러와의 관계

스프링 배치는 `언제 돌릴지`를 책임지는 제품이 아니다.

보통 아래처럼 역할을 나눈다.

- 스케줄러: 매일 새벽 2시에 시작
- 스프링 배치: 실제 데이터 처리 수행

함께 쓰는 예:

- cron
- Quartz
- Kubernetes CronJob
- 클라우드 스케줄러

## 실무 감각으로 정리하면

- `Job`: 큰 업무 단위
- `Step`: 세부 단계
- `Reader/Processor/Writer`: 읽기, 가공, 저장
- `Chunk`: 대량 데이터 묶음 처리
- `Tasklet`: 짧고 단일한 작업
- `ExecutionContext`: 진행 상태 메모
- `skip`: 문제 데이터는 넘기기
- `retry`: 일시적 오류는 재시도

## 한 줄 결론

스프링 배치는 `작업 단위`, `처리 단계`, `실패 복구`, `운영 기록`을 명확하게 나눠 대량 처리를 안전하게 만드는 프레임워크다.
