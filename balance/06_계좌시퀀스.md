# 계좌 상태 변화 상세 시퀀스

> 주문 → 체결 → 결제까지 계좌 내부 필드들이 어떻게 변하는지 단계별 추적

## 1. 계좌의 핵심 필드

시퀀스를 이해하기 전에, 계좌에서 추적하는 핵심 필드를 먼저 정리합니다.

| 필드명 | 설명 | 변동 시점 |
|--------|------|-----------|
| `cash_balance` | 확정된 현금 (D+0) | 입금, 결제 완료 시 |
| `pending_settlement` | 미결제 금액 (D+1, D+2) | 체결 시 증가, 결제 시 해소 |
| `locked_margin` | 잠긴 증거금 | 주문 시 증가, 체결/취소 시 해소 |
| `orderable_amount` | 주문 가능 금액 | 실시간 계산 |
| `withdrawable_amount` | 출금 가능 금액 | 결제 완료 시 증가 |
| `stock_holdings` | 보유 주식 | 결제 완료 시 변경 |

## 2. 주문 시스템 참여자와 책임

### 참여자(Participant) 정의

| 참여자 | 역할 | 취소 책임 |
|--------|------|-----------|
| **투자자 (User)** | 주문 발생, 취소 요청 | 자발적 취소 요청 |
| **주문시스템 (Order)** | 주문 접수, 라우팅, 상태 관리 | ❌ 취소 발생시키지 않음 |
| **계좌원장 (Ledger)** | 잔고 확인, 증거금 관리 | 증거금 부족 시 주문 거부 |
| **체결엔진 (Matching)** | 매수-매도 매칭 | 미체결 주문 만료 처리 |
| **거래소 (Exchange)** | 시장 운영, 체결 확정 | 장 마감 시 미체결 자동 취소 |
| **예탁결제원 (KSD)** | 결제, 정산 | 결제 실패 시 거래 무효화 (극히 드묾) |

### 취소 발생 주체별 구분

```plantuml
@startuml
skinparam backgroundColor white

rectangle "취소 유형" {
  
  rectangle "투자자 요청 취소" as user_cancel #LightGreen {
    note right: 미체결 상태에서\n투자자가 직접 취소
  }
  
  rectangle "시스템 자동 취소" as auto_cancel #LightYellow {
    note right
      - 장 마감 (15:30)
      - IOC/FOK 조건 미충족
      - 유효기간 만료
    end note
  }
  
  rectangle "거부 (Reject)" as reject #LightCoral {
    note right
      - 증거금 부족
      - 보유수량 부족
      - 호가 범위 초과
    end note
  }
}

@enduml
```

### 핵심 원칙
> **주문시스템(Order)은 취소를 "발생시키지" 않는다.**
> 
> - 주문시스템은 취소 **요청을 전달**하거나 **거부 결과를 반환**할 뿐
> - 취소의 **원인 제공자**: 투자자, 거래소, 계좌원장(잔고부족)
> - 취소의 **실행자**: 체결엔진 또는 거래소

## 3. 매수 주문 상세 시퀀스 (취소 포함)

### 시나리오
- 초기 예수금: 100만원
- 매수 주문: 100만원어치 (증거금률 40%)

```plantuml
@startuml
skinparam backgroundColor white
skinparam sequence {
  ArrowColor DarkSlateGray
  LifeLineBorderColor DarkSlateGray
  ParticipantBackgroundColor LightYellow
}

title 매수 주문 시퀀스 (100만원, 증거금률 40%)

actor 투자자 as user
participant "주문시스템" as order
participant "계좌원장\n(Account Ledger)" as ledger
participant "체결엔진" as matching
participant "예탁결제원\n(KSD)" as ksd
database "계좌 DB" as db

== 초기 상태 ==
note over db
  cash_balance: 1,000,000
  locked_margin: 0
  pending_settlement: 0
  orderable: 1,000,000
  withdrawable: 1,000,000
end note

== T일: 주문 접수 ==
user -> order: 매수 100만원 주문
order -> ledger: 증거금 확인 요청
ledger -> db: 조회
db --> ledger: orderable >= 400,000 ✓

ledger -> db: **증거금 잠금**
note right of db
  locked_margin: 0 → 400,000
  orderable: 1,000,000 → 600,000
  withdrawable: 1,000,000 → 600,000
end note

order -> matching: 주문 전송

== T일: 체결 ==
matching -> matching: 매도 주문과 매칭
matching --> order: 체결 완료
order -> ledger: 체결 반영

ledger -> db: **체결 처리**
note right of db
  locked_margin: 400,000 → 0
  cash_balance: 1,000,000 → 600,000
  pending_settlement: 0 → 600,000
  (미결제 잔금 60만원 기록)
end note

ledger -> ksd: 체결 내역 전송

== T+1일: 검증 ==
ksd -> ksd: 거래 대조/검증
note over ksd: 매수-매도 내역 일치 확인

== T+2일: 결제 ==
ksd -> ledger: 결제 지시

ledger -> db: **최종 결제**
note right of db
  cash_balance: 600,000 → 0
  pending_settlement: 600,000 → 0
  stock_holdings: +1,000,000원어치
  orderable: 0
  withdrawable: 0
end note

ksd --> user: 주식 입고 완료

@enduml
```

### 단계별 계좌 스냅샷

| 단계 | cash_balance | locked_margin | pending | orderable | withdrawable |
|------|--------------|---------------|---------|-----------|--------------|
| **초기** | 1,000,000 | 0 | 0 | 1,000,000 | 1,000,000 |
| **주문 접수** | 1,000,000 | **400,000** | 0 | **600,000** | **600,000** |
| **체결** | **600,000** | 0 | **600,000** | 600,000 | 600,000 |
| **T+2 결제** | **0** | 0 | 0 | 0 | 0 |

## 4. 취소 시나리오별 상세 시퀀스

### 4.1 투자자 요청 취소 (미체결 상태)

> **책임 주체**: 투자자 → 주문시스템(전달) → 체결엔진(실행)

```plantuml
@startuml
skinparam backgroundColor white
skinparam sequence {
  ArrowColor DarkSlateGray
  ParticipantBackgroundColor LightBlue
}

title 투자자 요청에 의한 주문 취소

actor 투자자 as user
participant "주문시스템" as order
participant "계좌원장" as ledger
participant "체결엔진" as matching
database "계좌 DB" as db

== 주문 접수 완료 상태 ==
note over db
  locked_margin: 400,000
  orderable: 600,000
  주문상태: "미체결"
end note

== 투자자 취소 요청 ==
user -> order: 주문 취소 요청
order -> order: 주문 상태 확인
note right of order
  상태: "미체결" ✓
  취소 가능
end note

order -> matching: 취소 요청 전달
matching -> matching: 주문장(Order Book)에서 제거
matching --> order: 취소 완료 응답

order -> ledger: **증거금 반환 요청**
ledger -> db: 증거금 해제
note right of db #PaleGreen
  locked_margin: 400,000 → 0
  orderable: 600,000 → 1,000,000
  withdrawable: 600,000 → 1,000,000
  주문상태: "취소완료"
end note

order --> user: 취소 완료 통보

@enduml
```

### 4.2 장 마감에 의한 자동 취소

> **책임 주체**: 거래소(시간 만료) → 체결엔진 → 주문시스템(통보)

```plantuml
@startuml
skinparam backgroundColor white
skinparam sequence {
  ArrowColor DarkSlateGray
  ParticipantBackgroundColor LightYellow
}

title 장 마감 시 미체결 주문 자동 취소

participant "거래소\n(Exchange)" as exchange
participant "체결엔진" as matching
participant "주문시스템" as order
participant "계좌원장" as ledger
database "계좌 DB" as db
actor 투자자 as user

== 15:20 - 동시호가 진입 ==
exchange -> matching: 동시호가 모드 전환

== 15:30 - 장 마감 ==
exchange -> matching: **장 마감 신호**
matching -> matching: 미체결 주문 일괄 취소
note over matching
  주문장에 남은 미체결 주문
  전량 "취소" 처리
end note

matching -> order: 취소 이벤트 전송
order -> ledger: 증거금 반환 요청
ledger -> db: 일괄 증거금 해제
note right of db #PaleGreen
  locked_margin: 400,000 → 0
  orderable: +400,000
  withdrawable: +400,000
  주문상태: "장마감취소"
end note

order --> user: [알림] 미체결 주문 자동 취소

@enduml
```

### 4.3 IOC/FOK 조건 미충족 즉시 취소

> **책임 주체**: 체결엔진(조건 판단) → 주문시스템(결과 통보)

| 조건 | 의미 | 취소 시점 |
|------|------|-----------|
| **IOC** (Immediate or Cancel) | 즉시 체결 가능한 수량만 체결, 나머지 취소 | 주문 즉시 |
| **FOK** (Fill or Kill) | 전량 체결 아니면 전량 취소 | 주문 즉시 |

```plantuml
@startuml
skinparam backgroundColor white

title IOC 주문 - 부분 체결 후 잔량 취소

actor 투자자 as user
participant "주문시스템" as order
participant "체결엔진" as matching
participant "계좌원장" as ledger
database "계좌 DB" as db

== IOC 주문 접수 ==
user -> order: 100주 IOC 매수
order -> ledger: 증거금 잠금
note right of db
  locked_margin: 400,000 (100주분)
end note

order -> matching: IOC 주문 전달

== 체결 시도 ==
matching -> matching: 매도 호가 확인
note over matching
  매도 가능 수량: 60주
  IOC 조건: 즉시 가능한 만큼만
end note

matching -> matching: 60주 체결, 40주 **즉시 취소**
matching --> order: 부분체결(60주) + 잔량취소(40주)

== 결과 반영 ==
order -> ledger: 체결 60주 + 취소 40주 반영
ledger -> db: 계좌 업데이트
note right of db
  체결분 증거금: 240,000 → pending
  취소분 증거금: 160,000 → **반환**
end note

order --> user: 60주 체결, 40주 취소 통보

@enduml
```

### 4.4 증거금 부족에 의한 주문 거부

> **책임 주체**: 계좌원장(잔고 검증) → 주문시스템(거부 반환)
> 
> ⚠️ 이것은 "취소"가 아니라 "거부(Reject)"

```plantuml
@startuml
skinparam backgroundColor white
skinparam sequence {
  ArrowColor DarkSlateGray
  ParticipantBackgroundColor LightCoral
}

title 증거금 부족 - 주문 거부 (Reject)

actor 투자자 as user
participant "주문시스템" as order
participant "계좌원장" as ledger
database "계좌 DB" as db

== 초기 상태 ==
note over db
  cash_balance: 300,000
  orderable: 750,000 (40% 증거금률)
end note

== 주문 접수 시도 ==
user -> order: 매수 100만원 주문
order -> ledger: 증거금 확인 요청

ledger -> db: orderable 조회
db --> ledger: 750,000

ledger -> ledger: 필요 증거금 계산
note over ledger
  필요: 1,000,000 × 40% = 400,000
  가용: 300,000
  **부족: 100,000**
end note

ledger --> order: ❌ 증거금 부족 (REJECT)
note right of order #Pink
  주문이 체결엔진에
  도달하지 않음
  = "취소"가 아닌 "거부"
end note

order --> user: 주문 거부 (증거금 부족)

note over db
  계좌 상태 변동 없음
  locked_margin: 0 (변동없음)
end note

@enduml
```

## 5. 취소 vs 거부 vs 체결실패 비교

| 구분 | 발생 시점 | 원인 | 증거금 | 계좌 영향 |
|------|-----------|------|--------|-----------|
| **거부 (Reject)** | 주문 접수 전 | 잔고/호가 검증 실패 | 잠기지 않음 | 없음 |
| **취소 (Cancel)** | 주문 접수 후, 체결 전 | 투자자 요청, 장마감, 조건 미충족 | 잠금 → 반환 | 원상복구 |
| **체결 (Fill)** | 매수-매도 매칭 시 | 정상 거래 | 잠금 → 정산 | 주식 이동 |
| **체결실패** | 결제 시 (T+2) | 미수금 | 부분 차감 | 반대매매 |

## 6. 취소 포함 전체 흐름 다이어그램

```plantuml
@startuml
skinparam backgroundColor white
skinparam activity {
  BackgroundColor LightYellow
}

title 주문 라이프사이클 (취소 분기 포함)

start

:주문 요청;

if (증거금 충분?) then (아니오)
  #LightCoral:주문 거부 (Reject);
  stop
else (예)
  :증거금 잠금;
  :주문 접수 완료;
endif

fork
  :투자자 취소 요청;
  :증거금 반환;
  #LightBlue:취소 완료;
  stop
fork again
  :체결 대기;
end fork

if (체결 성공?) then (아니오)
  if (장 마감?) then (예)
    :자동 취소;
    :증거금 반환;
    #LightBlue:장마감 취소;
    stop
  else (아니오)
    :계속 대기;
    note right: 다음 호가 매칭 시도
  endif
else (예)
  :체결 완료;
  :증거금 → 정산 전환;
endif

:T+1 검증;

:T+2 결제;

if (잔금 충분?) then (예)
  #PaleGreen:결제 완료;
  stop
else (아니오)
  #LightCoral:미수금 발생;
  :T+3 반대매매;
  stop
endif

@enduml
```

## 7. 매도 주문 상세 시퀀스

### 시나리오
- 초기: 주식 100만원어치 보유, 현금 0원
- 매도 주문: 100만원어치

```plantuml
@startuml
skinparam backgroundColor white
skinparam sequence {
  ArrowColor DarkSlateGray
  LifeLineBorderColor DarkSlateGray
  ParticipantBackgroundColor LightGreen
}

title 매도 주문 시퀀스 (100만원어치)

actor 투자자 as user
participant "주문시스템" as order
participant "계좌원장" as ledger
participant "체결엔진" as matching
participant "예탁결제원" as ksd
database "계좌 DB" as db

== 초기 상태 ==
note over db
  cash_balance: 0
  stock_holdings: 1,000,000
  pending_receive: 0
  orderable: 0
  withdrawable: 0
end note

== T일: 주문 접수 ==
user -> order: 매도 100만원 주문
order -> ledger: 보유 주식 확인
ledger -> db: 조회
db --> ledger: stock_holdings >= 1,000,000 ✓

ledger -> db: **주식 잠금**
note right of db
  stock_locked: 1,000,000
  (매도 대기 상태)
end note

order -> matching: 주문 전송

== T일: 체결 ==
matching -> matching: 매수 주문과 매칭
matching --> order: 체결 완료
order -> ledger: 체결 반영

ledger -> db: **체결 처리**
note right of db #LightBlue
  stock_holdings: 1,000,000 → 0
  pending_receive: 0 → 1,000,000
  (D+2 입금 예정)
  
  orderable: 0 → **1,000,000** ✓
  withdrawable: 0 (변동없음)
end note

ledger -> ksd: 체결 내역 전송

== T+1일: 검증 ==
ksd -> ksd: 거래 대조/검증

== T+2일: 결제 ==
ksd -> ledger: 결제 지시

ledger -> db: **최종 결제**
note right of db #PaleGreen
  pending_receive: 1,000,000 → 0
  cash_balance: 0 → 1,000,000
  
  orderable: 1,000,000
  withdrawable: 0 → **1,000,000** ✓
end note

ksd --> user: 매도 대금 입금 완료

@enduml
```

### 단계별 계좌 스냅샷

| 단계 | cash | stock | pending_receive | orderable | withdrawable |
|------|------|-------|-----------------|-----------|--------------|
| **초기** | 0 | 1,000,000 | 0 | 0 | 0 |
| **체결** | 0 | 0 | **1,000,000** | **1,000,000** | 0 |
| **T+2 결제** | **1,000,000** | 0 | 0 | 1,000,000 | **1,000,000** |

> 💡 **핵심 포인트**: 체결 직후 `orderable`은 즉시 증가하지만, `withdrawable`은 T+2 결제 후에야 증가

## 8. 매도 → 즉시 매수 시퀀스 (Same-day Trading)

### 시나리오
- A주식 100만원 매도 → 즉시 B주식 100만원 매수

```plantuml
@startuml
skinparam backgroundColor white

title 당일 매도 후 재매수 시퀀스

actor 투자자 as user
participant "계좌원장" as ledger
database "계좌 DB" as db
participant "예탁결제원" as ksd

== 초기 상태 ==
note over db
  cash: 0
  A_stock: 1,000,000
  B_stock: 0
end note

== T일 10:00 - A주식 매도 체결 ==
user -> ledger: A주식 매도
ledger -> db: 체결 처리
note right of db #LightBlue
  A_stock: 1,000,000 → 0
  pending_receive: +1,000,000
  
  orderable: 0 → 1,000,000
  withdrawable: 0 (변동없음)
end note

== T일 11:00 - B주식 매수 체결 ==
user -> ledger: B주식 매수 (100만원)
ledger -> db: 증거금 확인
note over db: orderable 1,000,000 >= 1,000,000 ✓

ledger -> db: 체결 처리
note right of db #LightYellow
  pending_receive: 1,000,000
  pending_pay: 1,000,000
  
  B_stock: 0 → 1,000,000 (예정)
  orderable: 1,000,000 → 0
  withdrawable: 0
end note

== T+2일: 결제 (상계처리) ==
ksd -> ledger: 결제 지시

ledger -> db: **네팅(Netting) 결제**
note right of db #PaleGreen
  pending_receive (A매도): 1,000,000
  pending_pay (B매수): 1,000,000
  ────────────────────
  차액: 0원 (상계 완료)
  
  A_stock: 0
  B_stock: 1,000,000 (확정)
  cash: 0
end note

ksd --> user: A→B 교체 완료

@enduml
```

### 상계(Netting) 처리 원리

```
T+2 결제 시점:
┌─────────────────────────────────────┐
│  받을 돈 (A매도):    +1,000,000원   │
│  줄 돈 (B매수):      -1,000,000원   │
│  ─────────────────────────────────  │
│  실제 이체 금액:            0원     │
└─────────────────────────────────────┘
→ 예탁결제원에서 자동 상계 처리
→ 실제 현금 이동 없이 주식만 교환
```

## 9. 미수금 발생 시퀀스

### 시나리오
- 예수금: 50만원
- 매수: 100만원 (증거금률 40% → 필요 증거금 40만원)
- T+2까지 추가 입금 없음

```plantuml
@startuml
skinparam backgroundColor white

title 미수금 발생 시퀀스

actor 투자자 as user
participant "계좌원장" as ledger
database "계좌 DB" as db
participant "예탁결제원" as ksd
participant "리스크관리" as risk

== 초기 상태 ==
note over db
  cash: 500,000
  orderable: 1,250,000
  (50만 ÷ 40% = 125만)
end note

== T일: 체결 ==
user -> ledger: 100만원 매수
ledger -> db: 체결 처리
note right of db
  cash: 500,000 → 100,000
  (증거금 40만 차감)
  pending_pay: 600,000
  (잔금 60만원 미결제)
end note

== T+1일: 입금 독촉 ==
ledger -> user: 잔금 60만원 입금 요청
user --> ledger: (입금 없음)

== T+2일: 결제 시도 ==
ksd -> ledger: 결제 지시
ledger -> db: 잔액 확인
note over db #Pink
  cash: 100,000
  필요액: 600,000
  부족액: **500,000**
end note

ledger -> risk: **미수금 발생 통보**
risk -> db: 미수금 기록
note right of db #LightCoral
  미수금(unsettled): 500,000
  계좌상태: "미수발생"
end note

== T+3일 09:00: 반대매매 ==
risk -> ledger: 반대매매 지시
ledger -> db: **강제 매도**
note right of db #Red
  보유주식 하한가 매도
  예상 회수: ~700,000
  (30% 하락 가정)
  
  미수금 상환: -500,000
  투자자 반환: ~200,000
end note

risk --> user: 반대매매 완료 통보
note over user #LightGray
  원금 100만원 중
  ~80만원 손실
end note

@enduml
```

## 10. 계좌 상태 전이 다이어그램

```plantuml
@startuml
skinparam backgroundColor white
skinparam state {
  BackgroundColor LightYellow
}

[*] --> 대기중: 입금

state "대기중" as idle {
  idle: cash > 0
  idle: orderable = cash
  idle: withdrawable = cash
}

idle --> 주문중: 매수 주문
state "주문중" as ordering {
  ordering: locked_margin > 0
  ordering: orderable 감소
}

ordering --> 대기중: 주문 취소
ordering --> 체결완료: 체결

state "체결완료" as filled {
  filled: pending_settlement > 0
  filled: 주식 예정 보유
}

filled --> 결제완료: T+2 결제
state "결제완료" as settled {
  settled: 주식 확정 보유
  settled: pending = 0
}

filled --> 미수발생: 잔금 부족
state "미수발생" as margin_call {
  margin_call: 미수금 > 0
  margin_call: 계좌 동결 위험
}

margin_call --> 결제완료: 추가 입금
margin_call --> 반대매매: T+3 미해결
state "반대매매" as liquidation {
  liquidation: 강제 청산
  liquidation: 손실 확정
}

liquidation --> 대기중: 정리 완료
settled --> 대기중: 매도

@enduml
```

## 11. 요약: 필드별 변동 타이밍

| 이벤트 | cash | locked | pending | orderable | withdrawable | stock |
|--------|------|--------|---------|-----------|--------------|-------|
| 매수 주문 | - | ↑ | - | ↓ | ↓ | - |
| 매수 체결 | ↓ | ↓ | ↑ | - | - | - |
| 매수 T+2 결제 | ↓ | - | ↓ | - | ↓ | ↑ |
| 매도 체결 | - | - | ↑(수취) | ↑ | - | ↓ |
| 매도 T+2 결제 | ↑ | - | ↓ | - | ↑ | - |
| **주문 취소** | - | **↓** | - | **↑** | **↑** | - |
| **주문 거부** | - | - | - | - | - | - |

## 12. 취소 책임 요약

```
┌──────────────────────────────────────────────────────────────┐
│                    취소/거부 책임 체계                        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [증거금 부족]                                               │
│       ↓                                                      │
│  계좌원장 → 주문시스템 → 투자자                              │
│  (검증실패)   (거부반환)   (거부통보)                         │
│                                                              │
│  [투자자 취소 요청]                                          │
│       ↓                                                      │
│  투자자 → 주문시스템 → 체결엔진 → 계좌원장                   │
│  (요청)    (전달)      (실행)     (증거금반환)                │
│                                                              │
│  [장 마감 / 조건 미충족]                                     │
│       ↓                                                      │
│  거래소/체결엔진 → 주문시스템 → 계좌원장                     │
│  (자동취소)        (통보)      (증거금반환)                   │
│                                                              │
└──────────────────────────────────────────────────────────────┘

💡 핵심: 주문시스템(Order)은 "전달자" 역할
   - 취소를 발생시키지 않음
   - 취소 요청을 라우팅하고 결과를 통보
```
