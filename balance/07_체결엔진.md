# 체결엔진 (Matching Engine)

> 주문을 받아 매수-매도를 매칭하고 체결을 확정하는 핵심 시스템

## 1. 체결엔진이란?

**체결엔진(Matching Engine)**은 증권 거래소의 심장부로, 투자자들의 매수/매도 주문을 받아 **가격-시간 우선 원칙**에 따라 매칭하고 거래를 성사시키는 시스템입니다.

### 핵심 역할

| 역할 | 설명 |
|------|------|
| **주문 접수** | 브로커(증권사)로부터 주문을 수신 |
| **주문장 관리** | Order Book에 미체결 주문을 정렬/보관 |
| **매칭 실행** | 매수-매도 주문을 규칙에 따라 매칭 |
| **체결 확정** | 매칭된 거래를 확정하고 결과 전파 |
| **미체결 처리** | 조건 미충족 주문 취소, 장 마감 처리 |

### 시스템 위치

```plantuml
@startuml
skinparam backgroundColor white

rectangle "투자자" as user
rectangle "증권사\n(Broker)" as broker #LightBlue
rectangle "주문시스템" as order
rectangle "체결엔진\n(Matching Engine)" as matching #LightYellow
rectangle "거래소\n(Exchange)" as exchange #LightGreen
rectangle "예탁결제원\n(KSD)" as ksd

user --> broker: 주문
broker --> order: 전달
order --> matching: 라우팅
matching --> exchange: 체결 보고
exchange --> ksd: 결제 지시

note right of matching
  **핵심 컴포넌트**
  - Order Book
  - Matching Algorithm
  - Trade Execution
end note

@enduml
```

## 2. Order Book (주문장)

### 구조

Order Book은 특정 종목에 대한 모든 미체결 주문을 가격별로 정렬한 데이터 구조입니다.

```plantuml
@startuml
skinparam backgroundColor white

title Order Book 구조 (삼성전자 예시)

rectangle "호가창 (Order Book)" {
  
  rectangle "매도 호가 (Ask/Offer)" #LightCoral {
    note as sell
      가격      수량      주문수
      ─────────────────────────
      71,000    500주     3건
      70,900    1,200주   8건
      70,800    800주     5건
      70,700    2,000주   12건  ← 최우선 매도호가
    end note
  }
  
  rectangle "스프레드" #LightGray {
    note as spread
      ────── 100원 ──────
    end note
  }
  
  rectangle "매수 호가 (Bid)" #LightGreen {
    note as buy
      70,600    1,500주   9건   ← 최우선 매수호가
      70,500    2,500주   15건
      70,400    1,800주   10건
      70,300    3,000주   20건
    end note
  }
}

@enduml
```

### Order Book 용어

| 용어 | 설명 |
|------|------|
| **Bid** | 매수 호가 (사고 싶은 가격) |
| **Ask/Offer** | 매도 호가 (팔고 싶은 가격) |
| **Best Bid** | 최우선 매수호가 (가장 높은 매수가) |
| **Best Ask** | 최우선 매도호가 (가장 낮은 매도가) |
| **Spread** | Best Ask - Best Bid (호가 갭) |
| **Depth** | 각 가격대의 주문 수량 |

### 데이터 구조 (개념적)

```
OrderBook {
  symbol: "005930"  // 삼성전자
  
  bids: [  // 매수 호가 (내림차순 정렬)
    { price: 70600, orders: [Order1, Order2, ...] },
    { price: 70500, orders: [Order3, Order4, ...] },
    ...
  ]
  
  asks: [  // 매도 호가 (오름차순 정렬)
    { price: 70700, orders: [Order5, Order6, ...] },
    { price: 70800, orders: [Order7, ...] },
    ...
  ]
}
```

## 3. Price-Time Priority (가격-시간 우선 원칙)

### 알고리즘 원리

체결엔진의 핵심 매칭 알고리즘은 **Price-Time Priority (FIFO)** 방식입니다.

```plantuml
@startuml
skinparam backgroundColor white

title Price-Time Priority 원칙

start

:새로운 주문 도착;

if (시장가 주문?) then (예)
  :최우선 호가와 즉시 매칭;
else (아니오)
  :지정가로 Order Book 진입;
endif

partition "매칭 우선순위" {
  :1. **가격 우선**
  매수: 높은 가격 우선
  매도: 낮은 가격 우선;
  
  :2. **시간 우선**
  동일 가격 내에서
  먼저 접수된 주문 우선;
}

:매칭 실행;

if (전량 체결?) then (예)
  :체결 완료;
else (아니오)
  :잔량 Order Book에 유지;
endif

stop

@enduml
```

### 예시: 매수 주문 매칭

```
현재 Order Book (매도 호가):
┌─────────────────────────────────────────┐
│  가격     수량    접수시간   주문번호   │
├─────────────────────────────────────────┤
│  70,700   500주   09:00:01   A001       │
│  70,700   300주   09:00:05   A002       │  ← 동일 가격, 시간 우선
│  70,700   200주   09:00:10   A003       │
│  70,800   1000주  09:00:02   A004       │
└─────────────────────────────────────────┘

새로운 매수 주문: 70,700원에 600주

체결 순서:
1. A001 전량 체결 (500주)
2. A002 부분 체결 (100주)
   → A002 잔량 200주는 Order Book에 유지
```

## 4. 주문 유형과 체결엔진 처리

### 주문 유형별 처리 방식

| 주문 유형 | 체결엔진 처리 | 미체결 시 |
|-----------|---------------|-----------|
| **시장가 (Market)** | 최우선 호가로 즉시 매칭 | 잔량 다음 호가로 계속 |
| **지정가 (Limit)** | 지정 가격 이상/이하에서만 매칭 | Order Book에 대기 |
| **IOC** | 즉시 가능한 수량만 체결 | 잔량 **즉시 취소** |
| **FOK** | 전량 체결 가능할 때만 체결 | 불가 시 **전량 취소** |
| **GTC** | 취소 전까지 유효 | 무기한 대기 |

### IOC/FOK 처리 시퀀스

```plantuml
@startuml
skinparam backgroundColor white

title IOC/FOK 주문 체결엔진 처리

participant "주문시스템" as order
participant "체결엔진" as matching
database "Order Book" as book

== IOC 주문 (100주) ==
order -> matching: IOC 매수 100주
matching -> book: 즉시 매칭 가능 수량 확인

alt 전량 매칭 가능
  matching -> matching: 100주 체결
  matching --> order: 체결완료
else 부분 매칭 가능 (60주)
  matching -> matching: 60주 체결
  matching -> matching: 40주 **즉시 취소**
  matching --> order: 부분체결 + 잔량취소
else 매칭 불가
  matching -> matching: **전량 즉시 취소**
  matching --> order: 전량취소
end

== FOK 주문 (100주) ==
order -> matching: FOK 매수 100주
matching -> book: 전량 매칭 가능 여부 확인

alt 전량 매칭 가능
  matching -> matching: 100주 체결
  matching --> order: 체결완료
else 전량 매칭 불가 (90주만 가능)
  matching -> matching: **체결 안함**
  matching -> matching: **전량 취소**
  matching --> order: 전량취소
end

@enduml
```

## 5. KRX EXTURE+ 시스템

한국거래소(KRX)의 체결엔진 시스템인 **EXTURE+**에 대해 알아봅니다.

### 시스템 개요

| 항목 | 내용 |
|------|------|
| **명칭** | EXTURE+ (EXchange Trading Using Reliable Engine) |
| **도입** | 2014년 (EXTURE 3.0), 2023년 고도화 |
| **플랫폼** | Linux 기반 |
| **처리속도** | 50-70μs (마이크로초) 레벨 |
| **처리용량** | 초당 수만 건 이상 |

### 아키텍처 특징

```plantuml
@startuml
skinparam backgroundColor white

title KRX EXTURE+ 아키텍처 개념도

rectangle "외부 접속" {
  rectangle "증권사 1" as broker1
  rectangle "증권사 2" as broker2
  rectangle "증권사 N" as brokerN
}

rectangle "KRX EXTURE+" #LightYellow {
  
  rectangle "Gateway Layer" as gw {
    note right: 주문 접수/검증\nProtocol 변환
  }
  
  rectangle "Order Management" as om {
    note right: 주문 상태 관리\n변경/취소 처리
  }
  
  rectangle "Matching Engine Core" as me #LightGreen {
    note right: **핵심 체결 엔진**\nPrice-Time Priority\n실시간 매칭
  }
  
  rectangle "Market Data" as md {
    note right: 시세 정보 생성\n호가창 배포
  }
}

rectangle "Post-Trade" {
  rectangle "체결 확인" as confirm
  rectangle "예탁결제원" as ksd
}

broker1 --> gw
broker2 --> gw
brokerN --> gw

gw --> om
om --> me
me --> md
me --> confirm
confirm --> ksd

@enduml
```

### 성능 최적화 기법

1. **In-Memory 처리**: Order Book을 메모리에 유지
2. **Lock-Free 알고리즘**: 동시성 최적화
3. **Event-Driven**: 비동기 이벤트 기반 처리
4. **Co-location**: 증권사 서버를 거래소 내 배치 허용

## 6. 체결엔진의 책임 범위

### 체결엔진이 **하는 것**

| 책임 | 상세 |
|------|------|
| Order Book 관리 | 미체결 주문 저장, 정렬, 검색 |
| 매칭 실행 | Price-Time Priority로 주문 매칭 |
| 체결 확정 | 매칭된 거래를 확정하고 ID 부여 |
| 조건부 취소 | IOC/FOK 미충족, 장 마감 시 미체결 취소 |
| 시세 정보 생성 | 체결가, 거래량, 호가창 정보 배포 |

### 체결엔진이 **하지 않는 것**

| 책임 없음 | 담당 시스템 |
|-----------|-------------|
| 증거금 확인 | 계좌원장 (Account Ledger) |
| 투자자 인증 | 증권사 (Broker) |
| 결제 실행 | 예탁결제원 (KSD) |
| 리스크 관리 | 리스크 시스템 |
| 투자자 취소 요청 접수 | 주문시스템 (Order) |

### 책임 경계 다이어그램

```plantuml
@startuml
skinparam backgroundColor white

title 시스템별 책임 경계

rectangle "투자자 영역" #LightBlue {
  (주문 발생)
  (취소 요청)
}

rectangle "증권사 영역" #LightGreen {
  (본인 확인)
  (주문 검증)
  (라우팅)
}

rectangle "거래소 영역" #LightYellow {
  rectangle "주문시스템" as order {
    (주문 접수)
    (상태 관리)
    (취소 전달)
  }
  
  rectangle "체결엔진" as matching {
    (Order Book)
    (매칭 실행)
    (체결 확정)
    (조건부 취소)
  }
  
  rectangle "계좌원장" as ledger {
    (증거금 확인)
    (잔고 관리)
  }
}

rectangle "결제 영역" #LightCoral {
  (T+2 결제)
  (주식 이전)
}

@enduml
```

## 7. 취소 발생과 체결엔진

### 체결엔진이 취소를 발생시키는 경우

```plantuml
@startuml
skinparam backgroundColor white

title 체결엔진이 취소를 실행하는 경우

start

partition "체결엔진 자체 판단에 의한 취소" {
  :IOC 주문 잔량 취소;
  note right: 즉시 체결 안 되면 바로 취소
  
  :FOK 주문 전량 취소;
  note right: 전량 체결 불가 시
}

partition "외부 신호에 의한 취소 실행" {
  :장 마감 신호 수신;
  note right: 거래소에서 15:30에 신호
  
  :미체결 주문 일괄 취소;
  
  :주문 유효기간 만료;
  note right: GTC 주문 등
}

:취소 결과를 주문시스템에 통보;

stop

@enduml
```

### 체결엔진이 취소를 실행만 하는 경우

```plantuml
@startuml
skinparam backgroundColor white

title 투자자 요청 취소 - 체결엔진 역할

actor 투자자 as user
participant "주문시스템" as order
participant "체결엔진" as matching
database "Order Book" as book

user -> order: 취소 요청
order -> order: 주문 상태 확인\n(미체결인지)

order -> matching: 취소 명령 전달
matching -> book: 주문 검색
book --> matching: 주문 발견

matching -> book: **주문 제거**
matching --> order: 취소 완료

note over matching
  체결엔진은 "실행자"
  취소 발생 원인은 투자자
end note

order -> order: 증거금 반환 처리
order --> user: 취소 완료 통보

@enduml
```

## 8. 체결 상세 시퀀스

### 매수-매도 매칭 과정

```plantuml
@startuml
skinparam backgroundColor white
skinparam sequence {
  ArrowColor DarkSlateGray
  ParticipantBackgroundColor LightYellow
}

title 체결엔진 내부 매칭 프로세스

participant "주문시스템" as order
participant "체결엔진" as matching
database "Order Book" as book
participant "시세배포" as md
participant "체결확인" as trade

== 신규 매수 주문 도착 ==
order -> matching: 매수 주문 (70,700원, 100주)

matching -> book: 최우선 매도호가 조회
book --> matching: 70,700원, 80주 (주문 A001)

matching -> matching: **매칭 판단**
note right
  매수가 70,700 >= 매도가 70,700
  → 매칭 성립
end note

matching -> matching: **부분 체결 처리**
note right
  A001: 80주 전량 체결
  신규 매수: 80주 체결, 20주 잔량
end note

matching -> book: A001 제거 (전량 체결)
matching -> book: 신규 매수 20주 추가

== 다음 매칭 시도 ==
matching -> book: 다음 최우선 매도호가 조회
book --> matching: 70,800원, 200주

matching -> matching: **매칭 판단**
note right
  매수가 70,700 < 매도가 70,800
  → 매칭 불성립
  → 잔량 20주는 Order Book에 대기
end note

== 체결 결과 배포 ==
matching -> trade: 체결 내역 전송
note right
  체결가: 70,700원
  체결량: 80주
  매수자: B001
  매도자: A001
end note

matching -> md: 시세 업데이트
note right
  최근 체결가: 70,700원
  거래량: +80주
  호가창 변동
end note

trade --> order: 체결 확인

@enduml
```

## 9. 장 운영과 체결엔진

### 장 시간별 동작 모드

| 시간 | 모드 | 체결엔진 동작 |
|------|------|---------------|
| 08:30-09:00 | 동시호가 (개장 전) | 주문 접수, 매칭 보류 |
| 09:00 | 개장 | 동시호가 일괄 체결 |
| 09:00-15:20 | 접속매매 | 실시간 연속 매칭 |
| 15:20-15:30 | 동시호가 (종가) | 주문 접수, 매칭 보류 |
| 15:30 | 장 마감 | 동시호가 체결 + 미체결 취소 |
| 15:40-16:00 | 시간외 단일가 | 단일가 매칭 |

### 동시호가 체결 원리

```plantuml
@startuml
skinparam backgroundColor white

title 동시호가 체결 (단일가 매매)

start

:08:30-09:00
주문 접수 (체결 없음);

:모든 주문을 Order Book에 축적;

:09:00 개장;

:균형가격 계산;
note right
  매수 누적 수량 = 매도 누적 수량
  최대 거래량 성립 가격 탐색
end note

:단일 가격으로 일괄 체결;
note right
  **예시**
  균형가: 70,650원
  체결량: 15,000주
  모든 체결이 같은 가격
end note

:미체결 주문은
Order Book에 유지;

:09:00:01~
접속매매(연속체결) 시작;

stop

@enduml
```

## 10. 체결엔진 장애 대응

### 장애 유형과 대응

| 장애 유형 | 영향 | 대응 |
|-----------|------|------|
| 체결엔진 지연 | 체결 속도 저하 | 큐잉, 재시도 |
| 체결엔진 중단 | 거래 중단 | 장 일시 정지, Failover |
| 데이터 불일치 | 호가창 오류 | 동기화, 재구성 |

### Failover 구조

```plantuml
@startuml
skinparam backgroundColor white

title 체결엔진 이중화 (HA 구조)

rectangle "Primary" #LightGreen {
  rectangle "체결엔진 A\n(Active)" as primary
}

rectangle "Secondary" #LightYellow {
  rectangle "체결엔진 B\n(Standby)" as secondary
}

database "공유 스토리지\n(Order Book 동기화)" as storage

rectangle "주문시스템" as order

order --> primary: 평상시
primary <--> storage: 실시간 동기화
storage <--> secondary: 실시간 복제

note bottom of secondary
  Primary 장애 시
  자동으로 Active 전환
  (Failover)
end note

@enduml
```

│     - 1순위: 가격 (유리한 가격 우선)                       │
│     - 2순위: 시간 (먼저 들어온 주문 우선)                  │
│                                                            │
