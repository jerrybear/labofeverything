# MySQL 삭제 후보 인덱스 고르는 기준

## 이 문서는 무엇을 설명하나

이 문서는 어떤 인덱스를 삭제 후보로 볼 수 있는지, 반대로 어떤 인덱스는 매우 조심해야 하는지를 기준 중심으로 정리합니다.

## 먼저 한 줄로 요약하면

- 삭제 후보 인덱스의 대표 신호는 `중복`, `낮은 사용률`, `과한 폭`, `읽기 이점 대비 큰 쓰기 비용`입니다.
- 반대로 `UNIQUE`, `FK`, 핵심 쿼리의 access path`, `covering 역할`이 있는 인덱스는 매우 조심해야 합니다.

## 1. 가장 먼저 보는 후보: 중복 인덱스

### 대표적인 삭제 후보

- 완전히 같은 컬럼 조합의 중복 인덱스
- 더 긴 복합 인덱스의 left-prefix로 충분히 대체되는 단일 인덱스

예시:

```sql
INDEX idx_a (user_id)
INDEX idx_b (user_id, created_at)
```

이 경우 `idx_a`는 삭제 후보가 될 수 있습니다.

### 단, 바로 삭제하면 안 되는 이유

- 짧은 인덱스가 더 작아 optimizer가 선호할 수 있음
- 정렬이나 covering 용도로 역할이 다를 수 있음

## 2. 사용률이 매우 낮은 인덱스

### 삭제 후보 신호

- 장기간 관찰했는데 거의 읽히지 않음
- 특정 배치/관리 기능에서도 확인되지 않음
- application code 검토에서도 사용처가 희박함

### 조심해야 할 점

- 월말 배치
- 관리자 기능
- 장애 대응용 ad hoc query

이런 쿼리는 평소엔 거의 없지만, 실제로 필요할 수 있습니다.

## 3. 읽기 이점보다 쓰기 비용이 훨씬 큰 인덱스

### 이런 인덱스는 후보가 될 수 있음

- 테이블보다 인덱스 총 크기가 훨씬 큼
- wide index라서 유지 비용이 큼
- DML이 매우 많은 테이블에서 읽기 이점이 낮음

### 운영 해석

- `INSERT`/`UPDATE`/`DELETE`가 잦은 테이블에서는
- 읽기에서 거의 안 쓰는 인덱스 하나도 꽤 큰 부담이 될 수 있습니다.

## 4. low-selectivity 인덱스

### 대표 예시

- `status`
- `is_deleted`
- `gender`
- 카디널리티가 매우 낮은 플래그성 컬럼

이런 인덱스는 optimizer가 잘 안 쓸 수 있고, 유지 비용 대비 효용이 낮을 수 있습니다.

### 하지만 예외

- 다른 컬럼과 결합된 복합 인덱스에서는 충분히 유용할 수 있음
- partial filtering과 ordering이 같이 걸리는 쿼리에서는 의미가 있을 수 있음

## 5. covering 역할이 없는 넓은 복합 인덱스

### 후보가 될 수 있는 경우

- 컬럼 수가 많은 복합 인덱스
- 실제 주요 쿼리에서 잘 안 타는 인덱스
- covering 효과도 없고 정렬 효과도 없는 인덱스

이런 인덱스는 저장 공간과 쓰기 오버헤드는 큰데, 실사용 가치가 낮을 수 있습니다.

## 6. 절대 먼저 의심하지 말아야 할 인덱스

### 1) UNIQUE 인덱스

- 데이터 무결성을 보장함
- 삭제하면 중복 입력 허용 문제가 생길 수 있음

### 2) FOREIGN KEY 관련 인덱스

- 참조 무결성과 검증 성능에 직접 관련됨
- MySQL이 요구하는 인덱스일 수 있음

### 3) 핵심 API 쿼리가 타는 인덱스

- 로그인, 주문 조회, 결제, 검색 같은 핵심 경로

### 4) batch update/delete가 의존하는 인덱스

- 삭제 후 rows examined가 급증할 수 있음
- lock footprint가 커지고 replication lag까지 악화될 수 있음

## 7. 삭제 후보를 고를 때 좋은 질문

인덱스 하나마다 아래를 묻는 것이 좋습니다.

- 최근 충분한 기간 동안 실제로 얼마나 읽혔는가
- 이 인덱스가 아니면 어떤 인덱스를 대신 탈까
- 아예 scan으로 떨어질 가능성은 없는가
- UNIQUE/FK/PK와 연관돼 있지 않은가
- write-heavy 테이블에서 유지 비용이 큰가
- replica 쪽에서 더 중요한 역할을 하고 있지는 않은가

## 8. 실무에서 자주 나오는 삭제 후보 유형

### 유형 A: 완전 중복 인덱스

- 가장 먼저 검토할 만한 대상

### 유형 B: left-prefix로 대체 가능한 단일 인덱스

- 실제 쿼리 플랜 검증이 필요

### 유형 C: 예전에 만든 기능이 사라져 더 이상 안 쓰는 인덱스

- 코드/서비스 정리와 함께 제거 후보

### 유형 D: 너무 넓고 무거운데 사용률이 낮은 인덱스

- 저장 공간 회수와 DML 비용 절감을 동시에 기대할 수 있음

## 9. 삭제 후보로 보기 전에 반드시 확인할 red flag

- 핵심 쿼리의 `possible_keys`에 계속 등장함
- `EXPLAIN`에서 실제 선택되는 인덱스임
- UNIQUE/FK 의존성이 있음
- 삭제 후 더 넓은 scan이 예상됨
- replica read나 apply에서 더 중요할 수 있음

## 10. 한 줄 결론

- 좋은 삭제 후보 인덱스는 "읽기 가치는 낮고 쓰기/공간 비용은 큰 인덱스"입니다.
